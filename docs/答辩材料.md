# DocAI 智能文档处理系统 — 答辩材料

> 本文档为项目答辩准备材料，涵盖技术架构、功能实现、创新要点、常见问答等内容。

---

## 一、项目概述

### 1.1 项目背景与问题

在日常办公中，工作人员需要从大量格式不一的文档资料中人工阅读、提取关键信息并手动整理到 Excel 表格或数据库。这一过程：
- 重复性高、操作繁琐
- 容易因疲劳导致疏漏和错误
- 严重消耗本应用于核心业务的时间和精力

### 1.2 解决方案

DocAI 是一款基于大语言模型（LLM）的智能文档处理系统，实现：
- **文档智能编辑交互**：通过自然语言指令与 AI 对话，完成文档分析、编辑、润色
- **非结构化文档信息提取**：自动识别并提取文档中的关键字段和实体信息
- **表格自定义数据填写**：从已提取的数据中自动匹配并填充模板表格

### 1.3 核心指标

| 指标 | 目标值 | 实现方案 |
|------|--------|---------|
| 信息提取准确率 | >80% | 大模型结构化抽取 + 字段标准化 |
| 表格填充准确率 | >90% | 规则优先 + LLM辅助 + 多级置信度 + 数据去重 + 统计聚合 + 无效值过滤 |
| 单模板响应时间 | <60秒 | 预提取架构 + 30秒填充上限 + directTableCopy直写 + 统计聚合快速路径 |

## 三、三大核心模块详解

### 3.1 模块一：文档智能编辑操作交互

**功能描述**：基于自然语言处理与文档结构理解技术，用户通过自然语言指令与 AI 对话，实现文档分析、编辑、润色、格式调整、内容提取等操作。支持在线修改文档内容并下载导出。

**技术实现**：
- 前端使用 SSE (Server-Sent Events) 实现流式响应，打字效果实时展示
- 支持关联已上传文档进行深度分析
- 支持多轮对话，对话历史持久化存储
- **AI在线文档编辑**：用户提出修改需求（如"删除第三节"、"将标题改为XXX"），AI自动读取原文档，输出修改后的完整内容，并自动保存为可下载文件
- 编辑意图检测：通过关键词匹配（修改/编辑/删除/增加/替换/润色等）自动识别
- AI 结果可导出为 Word/TXT/Markdown 文档
- 支持模型热切换（通义千问/DeepSeek等）

**API接口**：
```
POST /api/v1/ai/chat/stream  -- SSE流式AI对话
GET  /api/v1/ai/conversations -- 获取对话列表
POST /api/v1/ai/conversations -- 创建新对话
POST /api/v1/ai/documents/{docId}/apply-edit -- 文档在线修改
GET  /api/v1/ai/documents/edited/{fileName} -- 下载修改后文档
```

**关键代码设计**：
- SSE 流式传输避免长连接超时
- 5分钟超时保护机制
- 消息分为进度事件(progress)和完成事件(complete)
- 前端采用 AbortController 实现"停止生成"功能
- **文档在线编辑**：
  - 编辑意图关键词检测（修改/编辑/删除/增加/添加/替换/改为/改成/更新/移除/插入/润色）
  - AI 自动输出修改后的完整文档内容，后端自动保存至临时目录
  - 支持 docx（XWPFDocument）、txt、md 多格式输出
  - 前端检测到修改响应后弹出下载链接
  - 路径遍历防护：文件名校验禁止 `..` 等非法字符

### 3.2 模块二：非结构化文档信息提取

**功能描述**：支持 Word/Excel/TXT/Markdown 多格式文档解析，通过 AI 自动提取关键实体、数值、日期等结构化信息并存入数据库。

**技术实现**：
- 文档上传后立即返回，后台异步执行 LLM 提取
- 使用信号量(Semaphore)控制 LLM 并发数，避免 API 限流
- 带重试机制（最多3次），指数退避
- LLM 调用间隔控制（1.5秒），避免触发速率限制
- 前端轮询机制（前30秒每3秒，之后每8秒）检测提取完成状态

**提取字段规范**：
```json
{
  "fieldKey": "project_name",
  "fieldName": "项目名称",
  "fieldValue": "智能文档处理系统",
  "fieldType": "text",
  "aliases": ["课题名称"],
  "sourceText": "项目名称：智能文档处理系统",
  "confidence": 0.95
}
```

**字段标准化**：
- 内置 15+ 标准字段映射（project_name, owner, org_name, phone等）
- 日期统一为 YYYY-MM-DD
- 电话统一为纯数字
- 中文全角半角统一

**数据库表设计**：
- `source_documents` — 源文档记录
- `extracted_fields` — 提取字段
- `field_alias_dict` — 字段别名词典

### 3.3 模块三：表格自定义数据填写

**功能描述**：上传模板文件（Word/Excel），系统从已提取的文档数据库中自动搜索匹配数据并填充到模板对应位置。

**八阶段处理流程**：

| 阶段 | 名称 | 是否调用LLM | 说明 |
|------|------|------------|------|
| 0 | 基础设施 | 否 | 统一数据契约定义 |
| 1 | 源文档抽取 | 是(提取时) | 异步预提取，不在填表时执行 |
| 2 | 字段标准化 | 否 | 归一化处理 + 别名词典 |
| 3 | 模板解析 | 否 | 代码规则识别槽位 |
| 4 | 候选召回 | 否 | 五步规则召回 Top-5 |
| 5 | 难例判定 | 仅难例 | LLM从候选中选择 |
| 6 | 置信度融合 | 否 | 阈值决策 |
| 7 | 模板写回 | 否 | POI直接写回文件 |
| 8 | 结果审计 | 否 | 完整追溯日志 |

**候选召回策略（阶段4）**：
```
candidate_score = 
  0.40 * alias_match_score +    // 别名精确/模糊匹配
  0.20 * type_match_score +     // 字段类型匹配
  0.20 * source_vote_score +    // 多文档投票
  0.10 * context_match_score +  // 上下文关键字
  0.10 * field_confidence       // 提取置信度
```

**关键优化 — 多级去重机制**：
1. **字段级预去重**：填表前按 (fieldKey, fieldName, fieldValue) 三元组去重，保留置信度最高的记录
2. **候选级去重**：Top-5 候选按 fieldValue 去重，同一值只保留最高分
3. **行级去重**：directTableCopy 写回阶段跨多个源Excel去重，相同行数据只写一次

**统计聚合阶段**：
当模板中有汇总类槽位（如"病例数"、"总金额"等）且该标签同时出现在表头列中时，系统不再简单选取某一行的值，而是从源Excel读取该列所有数据，根据标签语义自动执行 SUM（求和）、COUNT（计数）、AVG（平均值）、MAX（最大值）、MIN（最小值）等统计运算，并将聚合结果填入模板。系统通过关键词检测自动识别聚合类型："总"→SUM、"平均"→AVG、"最大"→MAX、"最小"→MIN、"数量"→COUNT。

**header_below 智能路径分流**：
对于表头行模板（如统计表格），系统自动检测到 header_below 型槽位时，如果有源 Excel 文件可用，将跳过逐槽 LLM 调用，直接通过 directTableCopy 从源 Excel 复制匹配列的数据行到模板中，实测从38000+原始行去重为4500+有效行，处理时间约5秒。

**LLM难例判定条件（阶段5）**：
仅在以下条件满足时才调用LLM，否则纯规则决策：
1. Top1与Top2分差 < 0.05 且候选值不同
2. Top1分数 < 0.30 且别名分 < 0.2

**置信度阈值策略（阶段6）**：
- >= 0.70：直接填写（高置信）
- 0.05 ~ 0.70：填写，标记建议人工复核
- < 0.05：拒绝填写，保留空白（避免错填比少填更伤准确率）

**需求驱动精准填充（新增）**：
- 前端新增可选“用户需求输入框”，支持自然语言约束（时间、地点、日期、类型等）
- 候选召回新增 `requirement_match_score`，命中需求关键词显著加权，不命中显著降权
- 若仍存在空槽位，启用 `requirement_force_fill / mandatory_fallback` 做强兜底回填，尽量避免有源数据时空值返回

**在线预览（新增）**：
- 填表结果页新增“在线预览”Tab
- 基于 VueOffice 插件支持 `.docx` 与 `.xlsx` 在线预览
- 支持多结果文件切换查看，便于先验收后下载

**Excel模板写回（Apache POI XSSF）**：
- 定位 sheet + cell 坐标
- 只改目标值，不改整张 sheet 结构
- 遇到公式格不覆盖
- 支持合并单元格处理

**Word模板写回（Apache POI XWPF）**：
- 遍历表格定位目标单元格
- 空白单元格直接写值
- 内联模式（标签:____）定点替换
- 段落级替换保留原有格式

**超时控制**：
- 全局30秒上限（留充裕余量给写回和网络传输，总目标<60秒）
- header_below槽位有源Excel时跳过LLM逐槽决策，由directTableCopy直接处理
- 超时后跳过LLM调用，直接使用规则Top1结果
- 每次填表前自动清理旧的决策和审计记录，避免数据残留
- 无效值过滤：自动过滤占位符值（—、无、N/A、未知等），避免填入无效数据

## 四、创新要点（不超过3个）

### 创新点1：源文档预提取 + 模板实时匹配的双阶段架构

传统方案是在用户上传模板时才开始读取源文档并提取信息，导致响应时间长。DocAI 将信息提取和模板填充分离：
- **上传时抽取**：源文档上传后立即异步执行结构化抽取，不阻塞用户操作
- **填表时匹配**：模板上传后只做槽位识别和候选检索，从已建好的字段库中快速匹配

这种架构使得模板填充时不需要再次调用LLM读取源文档，响应时间从分钟级降至秒级。

### 创新点2：规则优先、模型辅助的多级决策机制

系统不直接让LLM决定最终填写值，而是采用分层决策：
1. **第一层 — 规则召回**：别名匹配、类型过滤、多文档投票
2. **第二层 — LLM判定**：仅对规则无法确定的难例调用LLM
3. **第三层 — 置信度兜底**：低置信度主动拒填，避免错误填写

这种设计确保了：
- 速度快（90%以上的字段纯规则决策，无需等待LLM）
- 准确率高（LLM只处理真正需要语义理解的字段）
- 结果可解释（每个字段都有决策原因和来源追溯）

### 创新点3：全链路可追溯的审计日志系统

每个填入字段都生成完整的审计记录，包含：
- 填入值及其来源文档和原文
- 所有候选字段及评分
- 决策方式（纯规则/规则+LLM/拒填）
- 决策理由（自然语言描述）

审计日志不仅支持结果验证，还为系统优化提供数据支撑。

## 八、常见问答准备

### Q1: 为什么选择 Apache POI 而不是 ExcelJS/python-docx？

A: 我们的后端主技术栈是 Java/Spring Boot，Apache POI 是 Java 生态中最成熟的 Office 文件操作库，可以直接在微服务中使用，无需跨语言调用。POI 的 XSSF（Excel）和 XWPF（Word）组件支持完整的读写操作，性能稳定。虽然参考方案建议用 Node.js + ExcelJS 和 Python + python-docx，但我们统一使用 Java 技术栈降低了系统复杂度和部署难度。

### Q2: 如何保证信息提取的准确性？

A: 我们采用多层保障：
1. **一文档一调用**：避免多文档混在一个prompt中导致跨文档信息污染
2. **结构化输出约束**：给LLM严格的JSON schema，确保输出格式规范
3. **字段带来源标注**：每个字段都记录来源文本和位置，支持验证
4. **字段标准化**：统一别名映射和格式归一化，减少后续匹配误差
5. **重试机制**：LLM调用失败自动重试（最多3次，指数退避）

### Q3: 如何实现 60 秒内完成表格填充？

A: 关键优化：
1. **预提取架构**：源文档信息在上传时异步提取，填表时无需再次调用LLM读取源文档
2. **规则优先**：90%+的字段通过规则召回直接决策，仅难例调用LLM
3. **超时控制**：30秒全局填充时限，超时后跳过LLM直接使用规则结果，留充裕余量给写回
4. **候选Top-K**：每个槽位只保留Top-5候选，LLM只做选择题不做开放式生成
5. **directTableCopy快速路径**：表头型模板自动检测并从源Excel直接复制数据，跳过逐字段LLM匹配
6. **多级去重**：字段级+候选级+行级三重去重，消除多源文档造成的重复数据
7. **无效值过滤**：前置过滤占位符值（"—"/"无"/"N/A"等），减少无效候选参与匹配的开销
8. **别名分数早期拒绝**：aliasScore < 0.1的候选直接拒填，避免无意义的LLM调用
9. **持久化文件存储**：源文件保存到 local-oss 持久目录，避免临时文件被系统清理导致二次读取失败
10. **实测表现**：普通模板约15-25秒完成，大数据量表头模板约30秒完成，均在60秒限制内

### Q4: LLM 用的是哪个模型？为什么？

A: 使用阿里云通义千问（qwen-plus）：
- 支持超长上下文（100万token），适合处理大文档
- DashScope 提供稳定的 API 服务
- 支持结构化JSON输出
- 国内部署，延迟低、合规
- 同时支持 DeepSeek 等模型热切换

### Q5: 如果 AI 提取的信息有误怎么办？

A: 系统有多层容错：
1. **置信度机制**：低置信度字段标记"建议人工复核"或直接拒填
2. **审计日志**：每个字段记录完整决策过程，便于人工检查
3. **前端决策详情**：填表结果页展示每个字段的置信度、决策方式和原因
4. **结果图表分析**：ECharts 可视化展示置信度分布、决策方式分布

### Q10: 模板槽位是如何识别的？

A: 完全使用代码规则识别（不调用LLM），三种模式：
1. **左标签右空白**：标签在左边，右边单元格为空
2. **上标签下空白**：标签在上面，下面单元格为空
3. **内联标签**：单元格内 "字段名：____" 格式

Word 使用 Apache POI XWPF 遍历所有表格、行、列。

### Q12: 候选字段召回的五步策略具体是什么？

A:
1. **精确别名匹配**（权重0.40）：槽位标签与字段名或别名词典精确匹配
2. **类型匹配**（权重0.20）：槽位期望类型与字段类型一致
3. **多文档投票**（权重0.20）：同一值在多个文档中出现则得分更高
4. **上下文关键字加权**（权重0.10）：槽位上下文包含相关关键字
5. **字段置信度**（权重0.10）：原始提取置信度

### Q13: 如何处理多个源文档中出现的重复数据？

A: 系统实现了三级去重机制：
1. **字段级预去重**：填表前按 (fieldKey, fieldName, fieldValue) 三元组去重，同一字段只保留置信度最高的记录
2. **候选级去重**：Top-5候选按 fieldValue 去重，同一值只出现一次，避免一个槽位被同一值的不同来源重复占据
3. **行级去重**：directTableCopy 写回阶段，跨多个源Excel文件使用行签名(所有列值拼接)去重，确保模板中不出现重复数据行

### Q14: directTableCopy 是什么？为什么需要这个优化？

A: directTableCopy 是针对"表头式"Excel模板的快速填充路径。当模板是类似统计报表（表头+数据行）的结构时：
- 系统检测到 header_below 类型的槽位
- 跳过逐槽调用LLM的慢路径
- 直接将源Excel文件中与模板表头匹配的列数据批量复制到模板中
- 实测38000+行原始数据去重后4500+有效行，耗时仅约5秒
- 大幅缩短了大数据量模板的填充时间

### Q15: 规则4"表头行检测"如何工作？

A: 在Excel模板解析阶段，扫描每一行：
- 统计该行非空单元格中有多少是"标签"（短文本、非纯数字、非说明性内容）
- 如果标签数 >= 2 且超过非空单元格的50%，则认定为表头行
- 在表头下方一行为每个标签创建 header_below 类型槽位
- 写回时由 directTableCopy 处理，支持批量数据填充

### Q18: 你们的项目和市面上的RPA工具（如UiPath、影刀）有什么区别？

A: 主要区别在于：
1. **理解能力**：RPA基于规则操作UI界面，我们基于LLM理解文档语义
2. **非结构化处理**：RPA难以处理格式不固定的文档，我们通过AI自动适应
3. **零模板配置**：RPA需要预先录制操作流程，我们的系统自动识别模板槽位
4. **可解释性**：每个填写值都有来源追溯和决策理由，RPA通常是黑盒操作

### Q19: 为什么不直接让大模型端到端完成填表？

A: 端到端方案有三个致命问题：
1. **速度慢**：大模型处理整个文档+模板需要几十秒甚至几分钟（超时）
2. **不可控**：大模型可能"幻觉"生成不存在的数据（准确率下降）
3. **不可追溯**：无法解释某个字段为什么被填入特定值
我们的分层方案让规则处理大部分字段（快且准），只在真正需要语义理解时才调用LLM（可解释），实现了速度和准确率的最佳平衡。

### Q20: 数据填充采用什么策略？

A: 系统采用**直接复制**策略——从源文档中找到与模板槽位匹配的字段，直接将原始数据复制到模板中，不做任何统计聚合或数值计算。具体流程：
1. 对于普通槽位（adjacent_blank），通过5维评分召回候选字段，取最高分候选的原始值直接填入
2. 对于表头类槽位（header_below），通过directTableCopy匹配源Excel列名，逐行复制所有数据
3. 所有决策都记录在审计日志中，支持完整的溯源
这种设计确保填充结果与源数据完全一致，避免了聚合计算可能带来的误差。

### Q21: 项目用到了哪些设计模式？

A:
1. **策略模式**：Excel和Word的槽位解析、写回分别实现不同策略
2. **责任链模式**：候选召回5步评分、多级决策（规则→LLM→兜底）
3. **观察者模式**：前端SSE实现流式AI响应
4. **模板方法模式**：文档提取的通用流程（读取→调用LLM→解析结果→入库）
5. **工厂模式**：LlmService支持多种AI Provider动态切换

### Q22: AI文档在线编辑是如何实现的？

A:
1. **意图检测**：用户消息中包含"修改/编辑/删除/增加/替换/润色"等关键词时自动识别为编辑请求
2. **文档读取**：后端读取源文档全文内容（docx/txt/md/xlsx），作为编辑上下文
3. **LLM编辑**：构建编辑Prompt，要求AI只输出修改后的完整文档内容
4. **自动保存**：检测到编辑意图且AI回复有效时，自动保存至临时目录（支持docx/md/txt格式）
5. **前端联动**：SSE响应携带`isModificationRequest`和`modifiedExcelUrl`，前端弹出下载链接
6. **手动触发**：用户也可通过"保存到文档"按钮主动调用apply-edit接口保存
7. **安全防护**：文件名校验防止路径遍历攻击

### Q23: 前端技术栈选型是什么？为什么选择这些？

A: Vue 3 + Element Plus + Vite + Pinia。Vue 3 的 Composition API 更适合复杂状态管理，Element Plus 提供成熟的企业级组件库，Vite 提供极快的 HMR 开发体验，Pinia 替代 Vuex 提供更简洁的状态管理。

### Q24: 后端技术栈选型是什么？

A: Java 17 + Spring Boot 3.2 + Spring Cloud Gateway + MyBatis-Plus + MySQL 8.4 + Redis 7.0 + Nacos 2.2。Spring Boot 3.2 是最新 LTS 版本，MyBatis-Plus 简化 CRUD，Nacos 统一配置管理。

### Q25: 为什么选择微服务架构而不是单体架构？

A: 项目包含用户认证、文件存储、AI 处理、API 网关等独立功能模块，微服务架构允许独立部署和扩展。AI 服务消耗资源较多，隔离后不影响其他服务。同时有利于团队分工协作。

### Q26: Gateway 网关的作用是什么？

A: Spring Cloud Gateway 承担统一入口、路由转发（StripPrefix=2）、跨域处理、JWT 鉴权过滤。所有 `/api/v1/**` 请求通过网关分发到对应微服务。

### Q27: JWT 认证是如何实现的？

A: 用户登录后签发 JWT Token（HS256 签名），前端存储在 localStorage。请求时在 Authorization Header 携带 Bearer Token。Gateway 全局过滤器验证 Token 有效性，白名单路径（登录/注册）跳过验证。

### Q28: 用户密码是如何存储的？

A: 使用 BCrypt 算法加盐哈希存储，不保存明文密码。登录时用 BCryptPasswordEncoder.matches() 验证。

### Q29: 文件上传的大小限制是多少？如何配置？

A: 默认 200MB，通过 `spring.servlet.multipart.max-file-size` 和 `max-request-size` 配置。Gateway 层也需要设置 `spring.cloud.gateway.httpclient.maxBodySize`。

### Q30: Nacos 在项目中的作用？

A: 统一配置中心，所有服务的数据库、Redis、AI API Key 等配置通过 Nacos 管理。支持运行时动态刷新配置，无需重启服务。

### Q31: Redis 在项目中用来做什么？

A: 缓存 JWT Token 黑名单（退出登录时标记）、缓存频繁查询的文档列表、限流计数器（控制 LLM API 调用频率）。

### Q32: SSE (Server-Sent Events) 和 WebSocket 有什么区别？为什么选 SSE？

A: SSE 是单向的服务端到客户端推送，基于 HTTP，实现简单。WebSocket 是双向全双工。AI 对话场景只需要服务端推流，SSE 更轻量、兼容性好、无需额外握手协议。

### Q33: 前端如何实现 SSE 流式消息接收？

A: 使用 fetch API 读取 Response.body 的 ReadableStream，通过 TextDecoder 解码，按 `\n\n` 分割事件，解析 `data:` 行获取 JSON 数据，逐字追加模拟打字效果。

### Q34: 如何实现"停止生成"功能？

A: 两阶段中止：1) 后端请求阶段使用 AbortController.abort() 中止 fetch 连接；2) 打字动画阶段停止定时器，立即展示已接收的完整内容。

### Q35: 对话历史如何管理？

A: 对话（conversations）和消息（messages）分表存储在 MySQL。每次新消息追加到对话下，支持上下文保持（最后5轮消息作为LLM输入）。前端通过轮询或页面加载时拉取。

### Q36: AI 模型切换是如何实现的？

A: 前端设置面板提供模型下拉选择（qwen-plus/qwen-turbo/deepseek-chat等），选择后存入 Pinia store 和 localStorage，每次 AI 请求在 body 中携带 model 参数，后端 LlmService 根据 model 字段动态选择 Provider。

### Q37: 文档提取为什么使用异步处理？

A: LLM 提取耗时 5-30 秒，同步会导致 HTTP 请求超时和前端无响应。异步模式：上传立即返回 parsing 状态，后台线程池执行提取，前端轮询状态直到完成。

### Q38: 提取时的信号量(Semaphore)是做什么的？

A: 限制同时提取的文档数为2个（Semaphore(2)），避免同时调用多个LLM请求导致 API 限流或超时。排队的文档等待许可释放后执行。

### Q39: 提取的重试机制是怎样的？

A: 最多重试3次，使用指数退避策略（第一次等1.5s，第二次等3s，第三次等6s），每次重试前检查API限流状态。

### Q40: 字段标准化映射是如何工作的？

A: 内置15+标准字段（project_name, owner, phone等），通过 normalizeLabel() 函数将中文标签映射到标准 fieldKey。例如"项目名称"、"课题名称"、"工程名称"都映射到 project_name。

### Q41: field_alias_dict 别名词典表有什么作用？

A: 存储用户自定义的字段别名映射（如"甲方"→"owner"），在候选召回阶段通过别名匹配提高字段对齐准确率，补充 normalizeLabel 内置映射的不足。

### Q42: 模板槽位（Slot）的四种检测规则是什么？

A:
1. **右空检测**：标签单元格的右侧为空 → 空格即为槽位
2. **下空检测**：标签下方为空 → 下方即为槽位
3. **内联检测**：单元格内包含"字段名：____" → 冒号后为槽位
4. **表头行检测**：整行多数为标签 → 下方行全部为 header_below 槽位

### Q43: 候选召回的五步评分权重是什么？

A: 别名匹配 40% + 类型匹配 20% + 多文档投票 20% + 上下文匹配 10% + 原始置信度 10%。综合得分 > 0.05 才入选候选。但即使入选，如果最佳候选的别名分数(aliasScore) < 0.1（即字段名完全不匹配），系统会直接拒填，避免将无关数据错误填入模板。

### Q44: 什么情况下会触发 LLM 难例判定？

A: 前提是 aliasScore >= 0.1（字段名有一定匹配度），在此基础上两个条件满足任一即触发：1) Top1和Top2候选分差 < 0.05 且值不同（难以区分）；2) Top1得分 < 0.30 且别名分 < 0.4（匹配程度中等，需AI确认）。同时不超时(30s)才实际调用。

### Q45: 置信度阈值为什么设 0.05？

A: 低于 0.05 的结果几乎确定是错误匹配，宁可留空让用户手动填写，也不填入错误数据。0.05-0.70 之间会标记“建议人工复核”。较低的拒填阈值确保了最大化填充率，同时结合无效值过滤（过滤占位符如“—”、“无”、“N/A”等）确保不填入垃圾值。

### Q46: 贪心兜底匹配是什么？

A: 当初始匹配完成后仍有空白槽位时启用（不再要求全部为空才触发）。按 normalizeLabel 和 fieldKey 精确匹配，然后模糊包含匹配，置信度固定 0.50，提供额外的填充保障。

### Q47: LLM 终极兜底是什么？

A: 贪心匹配后仍有空白槽位时，将源文档全部字段+原文内容（最长8000字符）一次性提交给 LLM，让其直接提取所有需填字段的值，置信度 0.65。这确保即使规则匹配完全失效，仍然能尽可能填充数据。

### Q48: directTableCopy 表格复制如何匹配列？

A: 系统在源Excel的前6行中搜索表头行（不限制必须在第0行），通过多级匹配找到最佳表头：精确匹配 → 忽略大小写 → 包含匹配 → normalizeLabel标准化匹配。选择匹配列数最多的行作为表头，然后从表头下一行开始逐行复制所有数据。当源文件被系统清理后，还会自动从OSS恢复文件再复制。

### Q49: 为什么不做统计聚合？

A: 系统设计原则是"原样复制"——用户明确要求直接将匹配的数据复制到模板，不做SUM/AVG等统计计算。这确保填充结果与源数据完全一致，避免计算误差，也更符合一般报表填写场景的需求。

### Q50: 填充审计日志记录了什么？

A: 每次填充的每个槽位决策详情：槽位标签、最终值、置信度、决策模式、候选列表、原因说明。审计批次ID关联同一次填表操作的所有决策。

### Q51: 前端的 ECharts 图表分析展示了什么？

A: 两个图表：1) 置信度分布饼图（高/中/低/未填写四档）；2) 决策方式分布饼图（规则匹配/AI提取/智能匹配/直接复制等各模式占比）。帮助用户快速评估填充质量。

### Q52: 文件存储方案是什么？

A: 采用持久化本地文件系统存储（data/local-oss/ 目录）。源文档和模板文件上传后直接保存到 local-oss 对应子目录（source_documents/、template_files/），不再使用Java临时目录，避免被系统清理。重名文件自动添加时间戳后缀区分。支持阿里云 OSS 对象存储扩展。当本地文件意外丢失时，填充逻辑会自动尝试从 OSS 恢复。

### Q53: 如何防止文件路径遍历攻击？

A: 上传时使用 Paths.get().getFileName() 提取纯文件名，过滤 `\`、`/`、`:` 等字符。目标目录通过 normalize() 确保路径合法。下载接口只允许访问指定目录内的文件。

### Q54: Docker Compose 编排了哪些服务？

A: MySQL 8.4（持久化数据存储）、Redis 7.0（缓存/限流）、Nacos 2.2（配置中心）、Nginx 1.24（前端静态托管+反向代理）。通过自定义网络 docai_network 互通。

### Q55: Nginx 在项目中的配置是什么？

A: 静态文件托管（前端 dist 目录），反向代理 `/api/` 请求到 Gateway（18080），WebSocket/SSE 长连接支持，GZIP 压缩优化传输。

### Q56: 数据库连接池用的什么？

A: HikariCP（Spring Boot 默认），配置最大连接数 10，连接超时 30s，空闲超时 600s。MyBatis-Plus 自动管理 Mapper 到连接的映射。

### Q57: MyBatis-Plus 相比原生 MyBatis 有什么优势？

A: 内置通用 CRUD（无需写 XML）、LambdaQueryWrapper 类型安全查询、自动填充（创建/更新时间）、逻辑删除、分页插件。节约 80% 的 Mapper 编写工作。

### Q58: 前端路由守卫是如何实现的？

A: Vue Router 的 beforeEach 全局前置守卫，检查 localStorage 中的 token 是否存在。未登录用户重定向到 /login 页。白名单路径（/login）无需验证。

### Q59: 前端状态管理用了哪些 Store？

A: documentStore（文档列表管理）和 index store（用户认证状态）。使用 Pinia defineStore 定义，支持持久化（localStorage）和 actions 异步操作。

### Q60: chatTaskManager 是做什么的？

A: 管理 AI 对话的异步任务队列，防止并发请求。维护当前活跃的 SSE 连接，支持中止正在进行的对话。

### Q61: 项目的错误处理策略是什么？

A: 后端统一异常处理（@ControllerAdvice），返回标准化 Result<T> 格式（code/message/data）。前端 Axios 拦截器统一处理 HTTP 错误，401 自动跳转登录。

### Q62: 如何实现前端文档列表轮询检测提取完成？

A: 上传后启动定时器：前30秒每3秒轮询一次文档状态，之后每8秒一次。检测到 uploadStatus 变为 parsed/failed 后停止轮询并更新UI。

### Q63: 跨域(CORS)是如何处理的？

A: Gateway 层配置全局 CORS（Access-Control-Allow-Origin: *，Allow-Methods: GET/POST/PUT/DELETE，Allow-Headers: Authorization/Content-Type），所有请求统一处理。

### Q64: 为什么用 Vite 不用 Webpack？

A: Vite 利用浏览器原生 ESM 实现毫秒级 HMR，开发体验远优于 Webpack。Rollup 构建产物更小。Vue 官方推荐的现代构建工具。

### Q65: 邮件发送功能如何工作？

A: 通过 JavaMail（Spring Boot Starter Mail）发送邮件，SMTP 配置通过环境变量注入（DOC_SMTP_USER/DOC_SMTP_AUTH_CODE）。支持发送 Excel 附件和 AI 生成内容到指定邮箱。

### Q66: 项目如何实现环境隔离？

A: 通过环境变量和 Nacos 配置隔离。本地开发连接本地 Docker 中间件，生产环境通过 Nacos 切换数据库/Redis/OSS 地址。敏感信息不硬编码到代码中。

### Q67: 项目的日志管理方案？

A: SLF4J + Logback，按服务分离日志文件（user-service.log/ai-service.log等），支持日志级别动态调整。关键操作（填表决策、AI 请求）记录 INFO 级别便于排查。

### Q68: 文档解析支持哪些格式？各用什么库？

A: Word(.docx) → Apache POI XWPFDocument；Excel(.xlsx) → Apache POI XSSFWorkbook；TXT → Files.readString；Markdown → Files.readString。PDF 暂不支持。

### Q69: 为什么不支持 PDF？

A: PDF 结构复杂（二进制流、表格识别难度大），Apache PDFBox 提取表格准确率低。后续可集成 Tabula-Java 或 OCR 增强。当前优先保证 Word/Excel 的高准确率。

### Q70: 多文档填表时如何处理字段冲突？

A: 同键同名同值的字段去重保留置信度最高的（field-level pre-dedup）；候选召回时按值去重保留最高分；多文档投票给频繁出现的值加分。最终由决策引擎综合裁定。

### Q71: 项目的部署方式？

A: 开发环境：start-lite-windows.ps1 一键全栈启动（Maven 构建 + Vite 构建 + Docker 中间件 + Java 服务）。生产环境：Docker Compose 编排全部服务，Nginx 反向代理。

### Q72: 如何保证服务启动顺序？

A: 启动脚本严格顺序：MySQL → Redis → Nacos → user-service → file-service → ai-service → gateway-service。每个服务启动后验证端口监听再启动下一个。

### Q73: 模板填充最终输出的文件格式？

A: 与模板原格式一致。Excel 模板(.xlsx) → 输出 _filled.xlsx；Word 模板(.docx) → 输出 _filled.docx。填充内容写入原模板的副本。

### Q74: 如何处理 Excel 合并单元格？

A: 解析阶段记录所有合并单元格的主单元格位置（mergedMainCells），槽位检测时跳过合并区域内的非主单元格，避免在合并区域内创建错误槽位。

### Q75: 如何处理 Excel 公式单元格？

A: 写回阶段检测单元格类型，如果是 CellType.FORMULA 则跳过不覆盖，保留原公式的计算功能。

### Q76: Word 文档填充如何定位表格位置？

A: 使用 XWPFDocument.getTables() 遍历所有表格，通过 position JSON（table/row/col 三元组）精确定位到目标单元格，写入决策值。

### Q77: 内联槽位(inline)的填充是怎样的？

A: 对于"字段名：____"格式的内联槽位，先读取单元格现有内容（标签部分），然后追加决策值到标签后面，而非覆盖整个单元格。

### Q78: 前端多模板批量填充是如何实现的？

A: 用户选择多个模板文件批量上传，前端遍历 templateFiles 数组，逐个执行：模板上传 → 解析槽位 → 自动填充 → 下载结果。所有模板共用同一批源文档。

### Q79: 填充结果如何提供给用户下载？

A: 填充完成后在服务端生成 _filled 文件，前端调用下载接口获取 Blob，使用 URL.createObjectURL 创建临时链接触发浏览器下载。

### Q80: 项目中有哪些安全措施？

A:
1. JWT 认证鉴权
2. BCrypt 密码加密
3. 文件路径遍历防护
4. SQL 注入防御（MyBatis-Plus 参数化）
5. XSS 防护（前端 DOMPurify 净化）
6. 文件类型白名单验证
7. 上传大小限制
8. Gateway 统一鉴权过滤

### Q81: DOMPurify 在前端的作用？

A: 对 AI 返回的 Markdown 渲染结果进行 HTML 净化，去除潜在的 XSS 攻击脚本（script 标签、事件处理器等），确保渲染安全。

### Q82: 前端 Markdown 渲染用了什么库？

A: marked.js 将 Markdown 转 HTML，配合 DOMPurify 净化，highlight.js 实现代码块语法高亮。

### Q83: 如何处理 LLM 返回的非标准 JSON？

A: extractJsonFromResponse() 函数用正则过滤 LLM 返回内容中的 Markdown 代码块标记（```json...```），提取纯 JSON 字符串后用 FastJSON 解析。

### Q84: 为什么选择 FastJSON 而不是 Jackson？

A: FastJSON 解析中文 JSON 性能更优，API 更简洁（JSONObject.parseObject 一行完成）。项目中大量中文字段名的 JSON 处理场景适合 FastJSON。

### Q85: 前端构建产物过大如何优化？

A: Element Plus 按需导入（减少包大小）、路由懒加载（动态 import）、Vite Tree-shaking 自动去除未使用代码。核心 JS 通过 GZIP 压缩到 375KB。

### Q86: 项目的测试策略？

A: 后端 JUnit 单元测试覆盖核心逻辑（候选召回、决策引擎），API 接口手动测试验证全流程，前端通过浏览器手动测试 UI 交互。后续可引入 Cypress E2E 测试。

### Q87: 如果 Nacos 宕机了会怎样？

A: 服务使用 CLI 参数启动（非 bootstrap.yml），配置已在启动时注入到 Spring 容器，Nacos 宕机不影响已运行的服务。新服务启动会失败，需重启 Nacos。

### Q88: 项目的监控方案？

A: 当前通过日志文件监控 + 端口健康检查。启动脚本自动验证各服务端口。后续可集成 Spring Boot Actuator + Prometheus + Grafana 实现可视化监控。

### Q89: 如何处理并发填表请求？

A: 每次填表操作独立事务，先清理旧数据再写入新数据。数据库层面通过 template_id 隔离，不同模板互不影响。同一模板并发填表通过数据库行锁防止脏写。

### Q90: 项目的数据库版本管理？

A: init.sql 脚本包含全部 DDL（CREATE TABLE IF NOT EXISTS），首次部署自动创建表结构。MySQL 容器挂载 init 目录实现自动初始化。

### Q91: 如何处理大文件上传？

A: Spring Multipart 配置 200MB 限制，Nginx 配置 client_max_body_size 匹配。大文件上传采用流式处理避免内存溢出，存储到本地磁盘后异步处理。

### Q92: 项目如何实现高内聚低耦合？

A: 微服务按功能域划分（用户/文件/AI/网关），服务间仅通过 HTTP API 通信。内部代码分层（Controller → Service → Mapper），依赖倒转通过接口注入。

### Q93: 阿里云 OSS 集成是如何实现的？

A: common-service 定义 OssService 接口，通过环境变量判断是否启用 OSS。启用时文件上传/下载走 OSS SDK，未启用时回退到本地磁盘存储。对上层透明切换。

### Q94: 为什么 AI 服务需要更大的 JVM 堆内存(768MB)？

A: AI 服务处理大型 Excel 文件时需要加载 XSSFWorkbook 到内存（Apache POI DOM 模式），35000+ 行数据的工作簿可能占用 200-500MB 内存。384MB 在极端情况下会 OOM。

### Q95: 如何优化 Excel 大文件处理的内存占用？

A: 当前使用 XSSFWorkbook（DOM 模式），后续可改用 SXSSFWorkbook（流式写入）或 SAX 事件模式（XSSFReader）来减少内存占用，支持百万级行数据处理。

### Q96: 前端如何实现文档预览？

A: Markdown/TXT 文档直接在浏览器内渲染（marked.js），Excel 数据展示在 el-table 中。Word/PDF 文件提供下载而非在线预览。

### Q97: 项目的版本发布流程？

A: Maven 统一版本号（1.0.0），修改代码后 `mvn clean package` 构建全部服务，`npm run build` 构建前端，拷贝 dist 到 Nginx，重启 Java 服务即可生效。

### Q98: 如何处理模板格式不规范的情况？

A: 四种槽位检测规则互补覆盖。如果自动识别失败，返回 0 个槽位并提示用户"模板未检测到可填充位置"。用户可调整模板格式后重试。

### Q99: 你在项目中遇到最大的技术挑战是什么？

A: 候选召回与决策引擎的准确率调优。初始版本只靠字段名精确匹配，填充率不到 30%。通过引入五步评分、别名词典、多文档投票、LLM难例判定、贪心兜底等多级策略，最终将填充率提升到 90%+。

### Q100: 项目还有哪些可以改进的地方？

A:
1. **PDF 支持**：集成 Tabula-Java 实现 PDF 表格提取
2. **流式 Excel 处理**：SXSSFWorkbook 支持百万级大文件
3. **E2E 自动化测试**：Cypress 或 Playwright 覆盖全流程
4. **容器化部署**：Java 服务也 Docker 化，K8s 编排
5. **实时协作**：WebSocket 支持多人同时编辑模板
6. **版本管理**：文档编辑历史记录与回滚
7. **国际化**：i18n 支持中英文切换

### 9.2 性能优化措施

| 优化项 | 具体措施 |
|--------|---------|
