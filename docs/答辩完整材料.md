## 一、项目概述

### 1.1 项目名称与定位

**DocAI — 基于大模型的智能文档信息提取与自动填表系统**

**一句话定位**：面向真实办公场景的智能文档自动化平台，解决"从非结构化文档到结构化表格"的全链路自动化问题。

**通俗理解**：想象你的桌上堆了100份合同（Word/Excel/TXT等各种格式），你需要把每份合同中的"甲方"、"乙方"、"金额"、"日期"等信息提取出来，再一一填入一张汇总表格。这个过程又慢又容易出错。DocAI 就是让 AI 帮你做这件事——它自动读取所有文档、提取关键信息、自动填入表格，整个过程不到30秒。

### 1.2 项目背景与痛点

在日常办公中，工作人员经常需要从大量格式不一的文档资料（Word、Excel、TXT、Markdown）中人工阅读、提取关键信息，再手动整理到 Excel 表格或数据库。

**具体场景举例**：
- 政府部门收到20份不同格式的工程合同，需要将每份合同的编号、金额、日期等信息整理成一张汇总表
- 教务处收到100份学生项目申报书，需要将每个项目的名称、负责人、指导老师等信息汇总
- 企业财务部每月要从大量票据中提取金额、日期、供应商等信息填入报表

这些场景的共同痛点：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **文档异构** | 同一批业务材料可能包含 Word/Excel/Markdown/纯文本，信息分散在正文、表格、附注、列表中 | 无法用单一工具统一处理 |
| **人工低效** | 需反复查找同类字段，同一信息被多次录入不同表格 | 某单位每月 80+ 小时重复劳动 |
| **质量不可控** | 人工疲劳导致 5%~15% 错误率，且无法追溯"这个值是从哪来的" | 数据可靠性无法保证 |

### 1.3 解决方案

DocAI 提供三大核心能力，覆盖文档处理的完整链路。用一个**流水线的比喻**来理解：

```
【原料】各种文档  →  【工序1】AI理解并提取关键信息  →  【工序2】信息存入数据库  →  【工序3】自动匹配并填入模板表格  →  【成品】填好的表格
```

三大核心模块分别对应：

1. **模块一：文档智能编辑交互**
   - **是什么**：像和 ChatGPT 聊天一样，你对 AI 说"帮我把第三段删掉"、"把标题改成XXX"，AI 自动修改文档内容并保存
   - **通俗理解**：类似于一个"AI文档秘书"，你口述修改要求，它自动执行
   - **核心技术**：SSE 流式传输（打字机效果实时展示AI回答）

2. **模块二：非结构化文档信息提取**
   - **是什么**：上传一份Word/Excel/TXT/Markdown文档，AI 自动识别并提取关键字段（项目名称、负责人、金额、日期等），存入数据库
   - **通俗理解**：类似于一个"AI阅读助手"，它读完整篇文档后告诉你"我找到了这些关键信息"
   - **核心技术**：多模态视觉语言模型（VL模型），不仅能读文字还能看图片

3. **模块三：表格自定义数据填写**
   - **是什么**：上传一个空白模板表格，系统从数据库中自动搜索匹配的数据并填充
   - **通俗理解**：类似于"AI填表员"，你给它一张空表格和一堆已提取的数据，它自动把对应的数据填到对应的格子里
   - **核心技术**：规则优先+LLM辅助的多级决策引擎（3800+行核心代码）

| 指标 | 目标值 | 实际表现 | 实现方案 |
|------|--------|---------|---------|

| 多端覆盖 | 4端 | 4端 | Web + Qt桌面端 + 微信小程序 + MCP协议服务 |

### 2.2 技术选型理由

> **答辩高频问题**："你们为什么选这个技术？"。以下每个选型都给出了"通俗理由"和"技术理由"两个层次的回答。

**为什么选 Vue 3 + Element Plus？**
- **通俗理由**：Vue 是国内最流行的前端框架，相当于前端开发的"普通话"——生态成熟、教程多、组件库丰富。Element Plus 是配套的"UI零件箱"，按钮、表格、弹窗、表单等常用组件开箱即用
- **技术理由**：Vue 3 的 Composition API 将逻辑关注点聚合（如 SSE 流式处理、文件上传状态管理），比 Options API 更适合复杂交互场景。Vite 构建工具提供毫秒级热更新(HMR)

**为什么选 Spring Cloud 微服务而非单体？**
- **通俗理由**：把一个大程序拆成几个小程序，各管各的事。好比开一家公司：与其让一个人又做销售又做财务又做技术，不如各设部门，出了问题也好定位
- **技术理由**：AI 服务消耗资源较多（Apache POI 加载大 Excel 可能需要 200-500MB 内存），隔离后不影响其他服务的稳定性。微服务架构也有利于团队分工和独立部署扩展

**为什么选 Apache POI 而非 ExcelJS/python-docx？**
- **通俗理由**：后端用 Java，那就选 Java 生态最好的 Excel/Word 读写库。就像你在中国买菜，用人民币就行，不需要换美元
- **技术理由**：统一 Java 技术栈避免跨语言 RPC 调用，Apache POI 的 XSSF（Excel）和 XWPF（Word）支持完整的读写操作，性能稳定

**为什么选 Qwen2.5-VL-7B-Instruct 作为核心模型？**
- **通俗理由**：这是阿里出的"既能读文字又能看图片"的AI模型，7B（70亿参数）大小适中——不太大（能在笔记本上跑），不太小（足够聪明），还支持微调
- **技术理由**：
  1. **多模态能力**：同时理解文本和图片，能识别文档中内嵌的表格截图、流程图等
  2. **结构化输出**：支持严格 JSON Schema 输出，适合信息提取任务
  3. **参数高效**：7B 参数约需 14-16GB 显存，M5 Max 48GB 统一内存完美适配
  4. **开源可微调**：支持 QLoRA 等参数高效微调，可针对特定文档领域优化

**为什么选 SSE 而非 WebSocket？**
- **通俗理由**：AI 聊天场景是"AI说，人听"的单向推送，不需要双向对话通道。SSE 就像看直播（单向接收画面），WebSocket 像打电话（双向通话）——直播就够了，不用打电话
- **技术理由**：SSE 基于标准 HTTP，实现简单、兼容性好、无需额外握手协议。WebSocket 的双向全双工对本场景是过度工程化

## 三、三大核心模块详解

### 3.1 模块一：文档智能编辑交互

#### 功能描述

**通俗理解**：就像和 ChatGPT 聊天一样，但不只是问答——你还可以让 AI 帮你修改文档。比如你说"帮我把第三段删掉"、"把标题改成智能文档系统"、"帮我润色一下第二段"，AI 会自动读取你的文档原文，修改后保存为新文件供你下载。

用户通过自然语言指令与 AI 对话，实现文档分析、编辑、润色、格式调整等操作。系统支持两种交互模式：

1. **纯对话模式**：用户直接提问，AI 回答。比如"帮我总结一下这份报告的核心内容"
2. **文档关联模式**：用户选择已上传文档，AI 读取文档全文后回答问题或执行修改。比如"把这份合同中的甲方名称改成XXX公司"

#### 技术实现细节

**SSE 流式传输机制**：
- 后端使用 Spring WebFlux 的 `SseEmitter` 发送流式响应
- 前端使用 `fetch` API 读取 `Response.body` 的 `ReadableStream`
- 通过 `TextDecoder` 解码，按 `\n\n` 分割 SSE 事件
- 解析 `data:` 行获取 JSON 数据，逐字追加实现打字机效果
- `AbortController` 实现"停止生成"功能：中止 fetch 连接 + 停止打字动画

**AI 在线文档编辑流程**：
```
用户输入修改指令 → 意图检测（关键词匹配）→ 读取源文档全文
→ 构建编辑 Prompt → LLM 输出修改后完整内容 → 自动保存到临时目录
→ SSE 返回修改标记 + 下载链接 → 前端弹出下载入口
```

意图检测关键词：修改、编辑、删除、增加、添加、替换、改为、改成、更新、移除、插入、润色

**多轮对话管理**：
- 对话（conversations）和消息（messages）分表存储
- 每次请求携带最近 5 轮消息作为上下文
- 支持模型热切换（qwen-plus / qwen-turbo / deepseek-chat）
- 前端 `chatTaskManager` 单例管理异步任务队列，防止并发请求冲突

**安全防护**：
- 文件名校验：禁止 `..`、`\`、`/`、`:` 等非法字符，防止路径遍历
- XSS 防护：AI 返回的 Markdown 渲染结果经 DOMPurify 净化
- 5 分钟超时保护：SSE 连接超时自动断开

#### API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/ai/chat/stream` | POST | SSE 流式 AI 对话 |
| `/api/v1/ai/conversations` | GET/POST | 获取/创建对话 |
| `/api/v1/ai/documents/{docId}/apply-edit` | POST | 文档在线修改 |
| `/api/v1/ai/documents/edited/{fileName}` | GET | 下载修改后文档 |

### 3.2 模块二：非结构化文档信息提取

#### 功能描述

**通俗理解**：你把一份合同上传到系统，AI 就像一个超快的阅读助手，几秒钟就能告诉你："我找到了——项目名称是XXX、甲方是XXX、金额是XXX万元、签订日期是2024年6月15日"。它不仅能读文字，还能看文档中嵌入的图片（比如表格截图）来提取信息。

支持 Word(.docx)、Excel(.xlsx)、TXT(.txt)、Markdown(.md) 四种格式文档解析，通过自部署的 Qwen2.5-VL-7B-Instruct 多模态视觉语言模型自动提取关键实体、数值、日期等结构化信息。

#### 为什么叫"非结构化"？

**结构化数据**：像 Excel 表格一样，数据在固定位置，有行有列有表头。程序直接按坐标读取即可。

**非结构化数据**：像一份合同文件，信息散布在文章各处——合同金额可能在第二段第三行，甲方名称可能在抬头，日期可能在文末。没有固定格式，每份文档的写法都不同。这就需要AI来"理解"文档内容后提取。

#### 核心流程

```
文档上传 → 立即返回(异步) → 文档解析(读取文本内容) → 图像提取(docx内嵌图片)
→ 构建多模态 Prompt(文本+图像) → 调用 VL 模型 → 解析 JSON 响应
→ 字段标准化(归一化+别名映射) → 结构化存入数据库 → 状态更新为 parsed
```

#### 技术实现细节

**异步提取架构**：
- 上传后立即返回 `uploading` 状态，后台线程池执行提取
- 使用 `Semaphore(2)` 控制并发，最多同时提取 2 个文档，避免 API 限流
- 带重试机制（最多 3 次），指数退避（1.5s → 3s → 6s）
- LLM 调用间隔 1.5 秒，避免触发速率限制
- 前端轮询：前 30 秒每 3 秒，之后每 8 秒

**异步提取的完整生命周期**：
```
用户点击"上传" → Controller接收文件
  → 保存到MinIO对象存储, 数据库记录状态=uploading
  → 返回HTTP 200（用户看到"上传中"）
  → 提交到 ExecutorService(4线程, 有界队列100)
  → Semaphore.acquire() 获取许可（最多2个并发）
  → 开始提取:
      1. 从MinIO下载文件到临时目录
      2. 根据后缀名选择解析器:
         - .docx → Apache POI XWPFDocument (提取段落+表格+图片)
         - .xlsx → Apache POI XSSFWorkbook (逐Sheet逐行读取)
         - .txt  → BufferedReader 逐行读取
         - .md   → 直接读取原文
      3. 拼接文本 + 提取图片(仅docx)
      4. 构建Prompt → 调用LLM → 解析JSON
      5. 若失败 → 重试(最多3次, 间隔1.5s/3s/6s)
      6. 若3次都失败 → 尝试规则提取兜底（正则匹配常见模式）
      7. 保存提取结果到数据库
      8. 更新状态: uploading → parsed (成功) 或 failed (失败)
  → Semaphore.release() 释放许可
```

**Prompt工程细节**（buildExtractionPrompt方法）：

Prompt分为三个部分拼接：
```
[系统指令] 你是一个专业的文档信息提取助手。请从以下文档内容中提取所有关键信息...
           必须以JSON数组格式返回，每个字段包含fieldKey/fieldName/fieldValue/fieldType/aliases/sourceText/confidence。
           注意：只提取文档中明确存在的信息，不要推断或编造。

[文档内容] ===文档内容开始===
           {实际文本内容，最长截取前12000字符}
           ===文档内容结束===

[图片内容] (仅docx且有内嵌图片时)
           以下是文档中的嵌入图片，请一并识别其中的信息：
           [图片1的Base64编码]
           [图片2的Base64编码]
```

**JSON解析容错**：LLM返回的JSON可能不规范，系统做了多层容错：
1. 尝试直接 `JSON.parse()`
2. 若失败，用正则提取 `[...]` 部分重试
3. 若仍失败，用正则提取 `{...}` 单个对象逐个解析
4. 最终兜底：启用规则提取（正则匹配"键：值"模式）

**多模态图像识别**（docx 专属）：
- 自动提取 docx 内嵌图片（`XWPFPictureData`），最多取 2 张
- 图片转 Base64 后与文本一起发送给 VL 模型
- VL 模型能识别图片中的表格、文字、图表等信息
- 这是纯文本模型无法实现的关键能力差异

**结构化输出约束**：
给 LLM 严格的 JSON Schema 约束，确保输出格式规范：
```json
{
  "fieldKey": "project_name",
  "fieldName": "项目名称",
  "fieldValue": "智能文档处理系统",
  "fieldType": "text",
  "aliases": ["课题名称"],
  "sourceText": "项目名称：智能文档处理系统",
  "confidence": 0.95
}
```

**字段标准化（normalizeLabel详细映射表）**：
- 内置 15+ 标准字段映射（project_name、owner、org_name、phone 等）
- `normalizeLabel()` 函数将中文标签映射到标准 fieldKey

| 标准key | 中文别名（任一命中即归一化） |
|---------|------------------------|
| project_name | 项目名称、课题名称、工程名称、项目名、课题名 |
| owner | 甲方、发包人、建设单位、业主、委托方 |
| contractor | 乙方、承包人、施工单位、中标单位 |
| org_name | 单位名称、公司名称、机构名称 |
| contact_person | 联系人、负责人、经办人 |
| phone | 联系电话、手机号码、电话号码、手机 |
| email | 电子邮箱、邮箱、email |
| address | 地址、住所、办公地址 |
| amount | 金额、合同金额、合同价款、总价 |
| sign_date | 签订日期、签署日期、合同日期 |
| start_date | 开始日期、起始日期、开工日期 |
| end_date | 结束日期、终止日期、竣工日期 |
| id_number | 身份证号、证件号码 |
| bank_account | 银行账号、开户行 |
| description | 项目描述、工程概况、项目简介 |

- 日期统一为 YYYY-MM-DD，电话统一为纯数字
- 中文全角半角统一转换

**一文档一调用原则**：每个文档独立调用一次 LLM，避免多文档混在一个 Prompt 中导致跨文档信息污染。

### 3.3 模块三：表格自定义数据填写（核心重点）

#### 功能描述

**通俗理解**：假设你有10份合同的提取结果（数据库里已经存好了每份合同的关键信息），现在你上传一个空白汇总表格模板（比如Excel模板，里面有"项目名称"、"甲方"、"金额"等空白格子），系统会自动把数据库中匹配的信息填进去。

**关键设计思想**：这个过程**不是**让 AI 从头读文档再填表，而是分两步：
1. **提前提取**：在你上传每份文档时，AI 就已经自动提取并存好了信息（模块二做的事）
2. **即时匹配**：上传模板时，系统只需要从数据库中快速搜索匹配的数据填入模板

这就像考试——你如果现场翻书找答案就很慢（端到端方案），但如果你提前做好了知识笔记，考试时翻笔记就很快（预提取+匹配方案）。

上传模板文件（Word/Excel），系统从已提取的文档数据库中自动搜索匹配数据并填充到模板对应位置。这是项目最复杂、代码量最大(3800+行)的模块。

#### 八阶段处理流程（通俗版）

> 用一个故事来理解八阶段：假设你是一个"AI填表员"，你的任务是把一堆简历信息填入汇总表。

| 阶段 | 名称 | 通俗解释 | 是否调用AI |
|------|------|---------|-----------|
| 0 | 基础设施 | "先定规矩"——约定好简历信息长什么样、表格格子长什么样、填写记录长什么样 | 否 |
| 1 | 源文档抽取 | "提前看完所有简历"——在每份简历上传时就已经提取并存好了信息，填表时不用再看 | 是(提前做) |
| 2 | 字段标准化 | "统一叫法"——"姓名"和"名字"其实是同一个东西，统一成一个标准名称 | 否 |
| 3 | 模板解析 | "看表格结构"——扫描这张汇总表有哪些空格子需要填、每个格子叫什么 | 否 |
| 4 | 候选召回 | "从笔记中找答案"——对每个空格子，从数据库中找出最可能匹配的5个候选值 | 否 |
| 5 | 难例判定 | "拿不准的问AI"——如果候选值之间差距很小（比如前两名分数几乎一样），才请AI帮忙选 | 仅~10%字段 |
| 6 | 置信度融合 | "下决定"——根据评分决定填还是不填。很有把握就填，一般把握也填但标记复核，没把握就不填 | 否 |
| 7 | 模板写回 | "动笔填"——用 Apache POI 把决定好的值写入 Excel/Word 文件 | 否 |
| 8 | 审计日志 | "记账"——每个格子填了什么、为什么填这个值、还有哪些候选、谁做的决定，全部记录 | 否 |

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 阶段0    │   │ 阶段1    │   │ 阶段2    │   │ 阶段3    │
│ 基础设施  │──▶│ 源文档   │──▶│ 字段     │──▶│ 模板     │
│ 数据契约  │   │ 预提取   │   │ 标准化   │   │ 解析     │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                   │
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 阶段7    │   │ 阶段6    │   │ 阶段5    │   │ 阶段4    │
│ 模板     │◀──│ 置信度   │◀──│ 难例     │◀──│ 候选     │
│ 写回     │   │ 融合     │   │ LLM判定  │   │ 召回     │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
      │

```

| 阶段 | 名称 | 是否调用LLM | 详细说明 |
|------|------|------------|---------|
| 0 | 基础设施 | 否 | 统一数据契约定义（ExtractedField、TemplateSlot、FillDecision） |
| 1 | 源文档抽取 | 是(上传时) | 异步预提取，不在填表时执行。填表时零等待 |
| 2 | 字段标准化 | 否 | normalizeLabel 归一化 + field_alias_dict 别名词典匹配 |
| 3 | 模板解析 | 否 | 纯代码规则识别槽位（四种检测模式，见下） |
| 4 | 候选召回 | 否 | 五维加权评分，为每个槽位召回 Top-5 候选 |
| 5 | 难例判定 | 仅难例 | 当规则无法确定时才调用 LLM（约 10% 的字段） |
| 6 | 置信度融合 | 否 | 三档阈值决策（高置信/建议复核/拒填） |
| 7 | 模板写回 | 否 | Apache POI 直接写回文件（XSSF/XWPF） |
| 8 | 结果审计 | 否 | 每个字段生成完整决策记录 |

#### 阶段3：模板槽位识别（四种检测模式）

**通俗理解**：系统需要先"读懂"模板表格的结构——哪些格子是标签（如"项目名称"），哪些格子是需要填数据的空位。就像你看一张填写好了标题的空白表格，一眼就能看出哪些地方该填。

全部使用代码规则识别，**不调用 AI**（速度极快）：

| 模式 | 名称 | 通俗理解 | 示例 |
|------|------|---------|------|
| 1 | 右空检测 | "标签在左边，右边的格子空着就是要填的" | `[项目名称][   ]` → 右边空格就是槽位 |
| 2 | 下空检测 | "标签在上面，下面的格子空着就是要填的" | 表头在上，数据行在下 |
| 3 | 内联检测 | "格子里写着'字段名：____'，横线处就是要填的" | `项目名称：________` |
| 4 | 表头行检测 | "一整行都是标签（列名）→ 这是统计报表的表头，下面所有行都是数据槽位" | 像"姓名|年龄|电话"这样的表头行 |

表头行判定规则：扫描每一行，统计非空单元格中"标签"（短文本、非纯数字、非说明性内容）的占比，若标签数 >= 2 且超过 50%，则认定为表头行。

**标签单元格判定逻辑（isLabelCell）**：一个单元格要被认为是"标签"，需要同时满足以下条件：
1. 非空（去除首尾空格后仍有内容）
2. 文本长度 ≤ 50 个字符（超过50字符更可能是描述性说明而非标签名）
3. 不是纯数字（如"12345"不是标签，而"项目编号"是标签）
4. 不在合并单元格的非主区域内（避免重复识别）

**合并单元格处理**：解析阶段首先收集模板中所有合并单元格的主单元格坐标（mergedMainCells集合），后续槽位检测会跳过合并区域内的非主单元格，避免在同一个合并区域内创建多个错误槽位。

**每个槽位的完整数据结构**：
```json
{
  "slotId": "slot_001",
  "label": "项目名称",
  "context": "项目名称：______",
  "position": "{\"sheet\":\"Sheet1\",\"row\":3,\"col\":2}",
  "expectedType": "text",
  "slotType": "right_blank",
  "requiredFlag": true
}
```

**Excel vs Word 槽位解析的差异**：

| 维度 | Excel (parseExcelSlots) | Word (parseWordSlots) |
|------|------------------------|----------------------|
| 定位方式 | sheet+row+col 坐标 | table+row+col 坐标 |
| 表头检测阈值 | 标签数≥2 且占非空50%+ | 非空≥3 且标签占80%+ |
| 合并单元格 | 需跳过非主区域 | Word表格无合并问题 |
| 扫描顺序 | 从左到右、从上到下 | 按表格索引、按行按列 |
| Pass数 | 单Pass（4规则并行） | 双Pass（先表头后常规） |

#### 阶段4：候选召回（五/六维加权评分）

**通俗理解**：对每个空格子，系统要从数据库中几百个字段里找出最匹配的。怎么找？用5（或6）个维度打分——名字像不像、类型对不对、有几份文档都出现了这个值、上下文相不相关、原始提取可信度、是否符合用户需求。每个候选值都综合打分，取前5名。

**打分公式（实际代码精确权重）**：

无用户需求时（5维）：
```
总分 = 0.34 × aliasScore     // 别名匹配分
     + 0.18 × typeScore       // 类型匹配分
     + 0.16 × voteScore       // 多文档投票分
     + 0.10 × contextScore    // 上下文匹配分
     + 0.08 × confScore       // 原始置信度
     + 0.14 × reqScore        // 需求匹配分（默认0，无影响）
```

有用户需求时（6维，权重重分配）：
```
总分 = 0.30 × aliasScore     // 别名权重降低（让位给需求）
     + 0.15 × typeScore
     + 0.13 × voteScore
     + 0.10 × contextScore
     + 0.08 × confScore
     + 0.22 × reqScore        // 需求权重显著提升
```

**各维度详细计算方式**：

**① aliasScore（别名匹配分，最关键维度）**——判断槽位标签和候选字段名是否指代同一事物

| 匹配等级 | 得分 | 判定条件 | 举例 |
|---------|------|---------|------|
| 精确匹配 | 1.0 | 字段名完全相同 | 槽位"项目名称" vs 字段"项目名称" |
| 标准化匹配 | 0.95 | normalizeLabel映射到同一标准key | 槽位"课题名" → project_name = 字段"项目名称" → project_name |
| 别名词典匹配 | 0.80~0.95 | field_alias_dict表中属于同一标准key | 词典记录：project_name → ["项目名称","课题名","工程名"] |
| 包含匹配 | 0.60 | A包含B或B包含A | 槽位"项目名"包含于字段"项目名称" |
| 部分重叠 | 0.35 | ≥3个汉字重叠且≥60%重叠率 | 槽位"联系电话号码" vs 字段"电话号码"（4字重叠） |
| 复合标签子项 | 0.55~0.85 | 拆分"A/B"后匹配子项 | 槽位"国家/地区" → 拆分为"国家"和"地区"分别匹配 |
| 无匹配 | 0.0 | 以上都不命中 | 槽位"项目名称" vs 字段"联系电话" |

**② typeScore（类型匹配分）**——判断候选值的数据类型是否符合槽位期望

| 情况 | 得分 | 说明 |
|------|------|------|
| 类型完全一致 | 1.0 | 槽位期望date，候选确实是date |
| 候选为text（通配） | 0.3 | text可以兼容大部分类型 |
| 期望phone但候选无数字 | -0.3 | 强类型不匹配，**负分惩罚** |
| 期望date但候选无日期特征 | -0.2 | 类型冲突惩罚 |
| 期望number但候选纯文本 | -0.2 | 类型冲突惩罚 |

**③ voteScore（多文档投票分）**——多份文档都提取出相同值则更可信
```
votes = 统计所有字段中fieldValue等于当前候选值的数量
maxVote = 所有字段中出现次数最多的值的票数
voteScore = votes / maxVote
```
例如：5份文档中有4份都提取到"张三"作为负责人 → voteScore = 4/4 = 1.0

**④ contextScore（上下文匹配分）**——槽位周围的文字是否与候选来源匹配
- slot.context包含候选的fieldKey → 0.7~0.8
- 候选的sourceText包含槽位label → 0.7
- 无上下文关联 → 0.0

**⑤ confScore（原始置信度）**——LLM提取时给出的信心
- 直接使用 field.confidence 值（0.0~1.0）
- 权重仅8%，作为参考因素

**⑥ reqScore（需求匹配分，仅当用户输入需求时生效）**——候选值是否符合用户约束
- 用户需求分词后，检查候选值/字段名是否包含需求关键词
- 命中多个关键词 → 分数累加
- reqScore < -0.50 且 aliasScore < 0.5 → **直接排除该候选**

**Top-5 候选去重**：多个候选如果 fieldValue 完全相同，只保留综合得分最高的那个。确保Top-5中每个值都是不同的。

**安全拒绝机制**：即使综合得分 > 0.05 入选候选，如果最佳候选的 aliasScore < 0.1（字段名完全不匹配），系统直接拒填。这就像即使考生总分过了线，但如果核心科目零分，也不能被录取。

#### 阶段5：LLM 难例判定（仅约 10% 字段触发）

**通俗理解**：90% 的格子靠规则打分就能确定填什么。但有些格子的前两名候选分数非常接近（比如一个85分一个83分），规则分不出高下，这时才请 AI 出马从候选中选一个。AI 只做"选择题"（从5个选项中选），不做"作文题"（不让它自由发挥），这样又快又可控。

**完整判定逻辑（伪代码）**：
```
needLlm = false

// 条件1：Top1与Top2近乎平手
IF candidates数量 >= 2 THEN
    IF (top1.score - top2.score < 0.05)              // 分差极小
       AND (top1.value ≠ top2.value)                 // 值确实不同
       AND (top1.aliasScore < 0.80)                  // 别名分不够碾压
    THEN needLlm = true

// 条件2：Top1本身就不够格
IF (top1.score < 0.30) AND (top1.aliasScore < 0.4)
THEN needLlm = true

// 条件3：别名高度匹配时规则已足够可靠
IF (top1.aliasScore >= 0.90)
THEN needLlm = false                                // 强制覆盖
```

**LLM 调用方式（callLlmJudge）**：
```
Prompt格式：
"模板槽位标签为'{slotLabel}'，上下文为'{context}'。
 以下是候选值列表：
 1. {value1}（来源：{doc1}，得分：{score1}）
 2. {value2}（来源：{doc2}，得分：{score2}）
 ...
 请选择最合适的候选编号，并给出0-1之间的置信度。
 仅返回JSON: {\"choice\": 1, \"confidence\": 0.85}"
```

AI只需从候选中选一个编号（选择题），不做开放式生成，避免幻觉。

**置信度融合**：如果LLM被调用，最终置信度按以下公式融合：
```
finalConfidence = 0.70 × candidate_score + 0.30 × llm_confidence
decisionMode = "rule_plus_llm"
```

**跳过LLM的快速决策路径**：
- aliasScore ≥ 0.90 → 直接使用Top1，decisionMode="rule_only"，置信度=candidate_score
- 超时保护：如果已消耗85%时间预算（≈46.75秒/55秒总预算），强制跳过LLM

#### 阶段6：置信度阈值策略

**通俗理解**：最终决定"填还是不填"——就像手术前的医生，非常有把握就直接做，有一定把握可以做但要告知风险，没把握就不做（错做比不做更危险）。

| 置信度区间 | 决策 | 通俗理解 |
|-----------|------|---------|
| >= 0.70 | 直接填写（高置信） | "我90%确定这是对的，直接填" |
| 0.05 ~ 0.70 | 填写，标记建议复核 | "我觉得大概率是对的，先填着但请人工看一眼" |
| < 0.05 | 拒绝填写，保留空白 | "我几乎确定这是错的，宁可留空也不填错" |

**为什么 < 0.05 就不填？**——因为在填表场景中，**错填比少填更危险**。留空格子用户一眼就能看到需要补充，但填了一个错误值用户可能不会注意到。

#### 多级兜底机制（三层递进策略）

**通俗理解**：就像考试填空题的答题策略——先用自己最有把握的知识答，答不出来看看题目有没有提示，还答不出来就翻笔记，实在不行就靠直觉猜一个。

当初始匹配完成后仍有空白槽位时，系统**依次执行**三层兜底，每层处理完后检查剩余空白数，全部填满则提前终止：

**第一层：贪心兜底匹配**（"翻笔记"策略）
```
FOR 每个空白槽位 DO
    step1: normalizeLabel(slotLabel) → 标准key
    step2: 在 fieldsByKey 中查找标准key
    IF 找到 THEN 填入最高置信度字段, mode="greedy_fallback", conf=0.50
    ELSE
        step3: 在 fieldsByName 中模糊查找（忽略大小写、包含匹配）
        IF 找到 THEN 填入, mode="greedy_fallback", conf=0.50
END FOR
```

**第二层：需求驱动填充**（"划重点"策略，仅当用户输入需求时执行）

用户可输入自然语言约束（如"只要2024年德州市的数据"），系统处理流程：
1. 对需求文本分词，提取有效关键词（长度≥2，过滤停用词）
2. 提取日期范围（如"2024-01-01 to 2024-12-31"）
3. 对仍为空白的槽位，重新执行候选召回，此时 reqScore 权重提升至 0.22
4. 命中需求关键词的候选大幅加分，不命中的降权
5. 若需求过滤后所有候选被排除（过于严格），自动放宽条件重试

**第三层：LLM 终极兜底**（"靠直觉猜"策略）
```
IF 仍有空白槽位 AND 剩余时间>5秒 THEN
    step1: 收集所有已提取字段的 sourceText + fieldName:fieldValue
    step2: IF 拼接文本 < 8000字符 THEN 追加源文档原文
    step3: 构建Prompt:
        "以下空白字段需要填写：[slotLabel1, slotLabel2, ...]
         已知数据：[所有字段信息]
         请直接以JSON格式返回每个字段应填的值。"
    step4: 解析LLM返回的JSON
    step5: 对每个匹配的字段: mode="llm_fallback", conf=0.65
END IF
```

**最终强制填充**（"宁猜不空"策略）：
- 如果经过三层兜底后仍有空白，且该槽位存在任何候选（哪怕分数很低），则取需求匹配度最高的候选强制填入
- 置信度：0.35~0.78（视候选质量而定）
- 标记原因为"强制兜底填充，建议人工复核"

#### directTableCopy 快速路径（关键优化）

**通俗理解**：当模板是那种"表头+数据"的统计报表（比如一个Excel表头写着"城市|GDP|人口|面积"，下面是空行要填数据），系统不会笨笨地一个格子一个格子找匹配，而是直接去源Excel中找到相同的列（也有"城市|GDP|人口|面积"），然后把整列数据批量复制过来。就像你抄表格——如果两张表结构一样，直接一列一列抄，比一格一格对快得多。

**触发条件判定**：
```
IF 模板存在 header_below 型槽位 THEN
    IF 源文档列表中包含至少一个 .xlsx 文件 THEN
        → 启用 directTableCopy 快速路径
    ELSE IF 源文档中有 docx/md/txt 且 allFields 非空 THEN
        → 退回 llmMultiRowFill() 路径（通过LLM从文本中提取表格数据）
    END IF
END IF
```

**详细处理流程**：

```
Step 1: 表头行搜索（不固定第0行）
    FOR 源Excel的每个Sheet DO
        FOR row = 0 TO 5 DO          // 扫描前6行
            统计当前行的非空单元格数
            IF 非空数 >= 2 THEN 记为候选表头行
        END FOR
    END FOR

Step 2: 多级列匹配（五级递进）
    FOR 模板表头的每个列名 tplLabel DO
        FOR 源Excel候选表头行的每个列名 srcLabel DO
            级别1: tplLabel == srcLabel                    → 精确匹配
            级别2: tplLabel.equalsIgnoreCase(srcLabel)     → 忽略大小写
            级别3: tplLabel.contains(srcLabel) 或反向      → 子串包含
            级别4: normalizeLabel(tplLabel) == normalizeLabel(srcLabel)  → 标准化
            级别5: 拆分 tplLabel 的 "/" 复合标签后逐子项匹配 → 复合匹配
        END FOR
    END FOR
    选择匹配列数最多的源表头行

Step 3: 需求过滤（可选）
    IF 用户输入了需求约束 THEN
        提取日期范围（如 2024-01-01 ~ 2024-12-31）
        提取关键词列表（长度≥2，过滤停用词）
        FOR 每一行源数据 DO
            IF 行中不包含任何关键词 OR 日期不在范围内
            THEN 跳过该行
        END FOR
        IF 过滤后行数 == 0 THEN 放弃过滤，使用全部行（防止过滤过严）
    END IF

Step 4: 逐行复制 + 行级去重
    rowSignatureSet = 空集合
    FOR 每一行源数据 DO
        signature = 所有列值拼接为字符串
        IF signature NOT IN rowSignatureSet THEN
            写入模板对应行
            rowSignatureSet.add(signature)
        ELSE
            跳过重复行
        END IF
    END FOR

Step 5: 更新决策记录
    mode = "direct_table_copy"
    confidence = 0.95
```

**实测效果**：38000+ 行原始数据去重为 4500+ 有效行，耗时仅约 5 秒。

**与逐槽LLM调用的性能对比**：
| 指标 | directTableCopy | 逐槽LLM调用 |
|------|----------------|------------|
| 处理38000行 | ~5秒 | 38000×1秒 ≈ 10小时 |
| LLM调用次数 | 0次 | 38000次 |
| API成本 | ¥0 | ¥380+ |
| 准确率 | 95%（列匹配正确则行正确） | ~88%（每次LLM调用都有幻觉风险） |

#### 三重去重机制

| 去重层级 | 时机 | 策略 |
|---------|------|------|
| 字段级预去重 | 填表前 | 按 (fieldKey, fieldName, fieldValue) 三元组去重，保留置信度最高 |
| 候选级去重 | Top-5 召回后 | 按 fieldValue 去重，同一值只保留最高分 |
| 行级去重 | directTableCopy 写回时 | 跨多源 Excel，相同行签名只写一次 |

**字段级预去重详解**：
```
输入: 10份文档 × 平均15个字段 = 150个字段
Step1: 按 fieldKey 分组 → Map<String, List<ExtractedField>>
Step2: 同一 fieldKey 内，按 fieldValue 分组
Step3: 每组只保留 confidence 最高的那个
输出: 去重后约 40~60 个独立字段
```
这一步在候选召回之前执行，大幅减少后续评分的计算量。

#### 模板写回详解（阶段7）

**Excel 写回（writeBackExcel）**：
```
Step 1: 从 MinIO 下载原始模板文件到临时目录
Step 2: 用 Apache POI XSSFWorkbook 打开模板
Step 3: FOR 每个填充决策 DO
    解析 position JSON → 定位到 sheet/row/col
    获取目标单元格 cell = sheet.getRow(row).getCell(col)
    IF cell 类型为 FORMULA THEN 跳过（不覆盖公式）
    IF cell 类型为 NUMERIC 且值为数字 THEN setCellValue(Double)
    ELSE setCellValue(String)
    保持原单元格样式不变（字体、边框、背景色等）
END FOR
Step 4: 写入到新文件（不覆盖原模板）
Step 5: 上传新文件到 MinIO → 返回下载URL
```

**Word 写回（writeBackWord）**——处理多表格的复杂情况：
```
Step 1: 打开 Word 模板 (XWPFDocument)
Step 2: 获取文档中的所有表格列表 doc.getTables()
Step 3: FOR 每个填充决策 DO
    解析 position → tableIndex/row/col
    获取目标表格 table = tables.get(tableIndex)
    获取目标单元格 cell = table.getRow(row).getCell(col)
    
    // 保持原格式写入
    IF cell 有已存在的段落和 Run THEN
        使用第一个 Run 的字体设置
        清空原内容 → 设置新值
    ELSE
        创建新 Run → 写入值
    END IF
END FOR
Step 4: 保存到新文件 → 上传 MinIO
```

**Word 多表格定位问题**：Word 文档可能包含多个独立表格（如第一页是基本信息表，第二页是财务信息表），系统通过 `tableIndex` 精确区分每个表格。解析阶段就记录了每个槽位属于第几个表格的第几行第几列，写回时按此定位。

#### 统计聚合

当模板中有汇总类槽位（"病例数"、"总金额"等）且该标签同时出现在源 Excel 表头列中时，系统从源 Excel 读取该列所有数据，自动执行聚合运算：

```
Step 1: 检测汇总关键词
    FOR 每个已填充的槽位 DO
        IF slotLabel 包含 ["总","合计","汇总","小计","累计","平均","最大","最小","数量","个数","人数"] THEN
            标记为汇总槽位
    END FOR

Step 2: 查找对应数据列
    在 directTableCopy 已复制的数据中，找到与汇总槽位同名（或别名匹配）的列

Step 3: 执行聚合
    - 标签含"总"/"合计"/"汇总"/"小计"/"累计" → SUM
    - 标签含"平均"/"均" → AVG
    - 标签含"最大"/"最高" → MAX
    - 标签含"最小"/"最低" → MIN
    - 标签含"数量"/"个数"/"人数"/"病例数" → COUNT

Step 4: 写入聚合结果, confidence = 0.90
```

#### 审计日志（阶段8）

每个槽位的填充结果都生成一条完整的决策记录（FillDecision），用于事后追溯和质量审计：

```json
{
  "slotId": "slot_001",
  "slotLabel": "项目名称",
  "filledValue": "智能文档处理系统",
  "confidence": 0.92,
  "decisionMode": "rule_only",
  "topCandidates": [
    {"value": "智能文档处理系统", "score": 0.92, "source": "合同.docx"},
    {"value": "DocAI系统", "score": 0.71, "source": "方案.docx"},
    {"value": "文档AI平台", "score": 0.45, "source": "简介.md"}

  "reviewFlag": false,

```

**审计记录的价值**：
- **可解释性**：用户可以看到"为什么填这个值"——因为哪个维度得了多少分
- **可追溯性**：每个值都能追溯到来自哪个源文档的哪段文字
- **可复核性**：标记了"建议复核"的字段，用户可以逐个检查决策依据
- **可改进性**：收集错误填充的审计日志，可用于后续优化评分权重

#### 超时控制

- 全局 30 秒上限（总目标 < 60 秒）
- header_below 槽位有源 Excel 时跳过 LLM，由 directTableCopy 直接处理
- 超时后跳过 LLM 调用，直接使用规则 Top1 结果
- 无效值过滤：自动过滤占位符值（"—"、"无"、"N/A"、"未知"等）

**超时保护的完整策略**：
```
fillStartTime = System.currentTimeMillis()
TOTAL_BUDGET = 55_000ms (55秒)

// 在每个槽位处理前检查
FOR 每个槽位 DO
    elapsed = now() - fillStartTime
    IF elapsed > TOTAL_BUDGET × 0.85 (≈46.75秒) THEN
        强制跳过LLM，使用规则Top1
    IF elapsed > TOTAL_BUDGET THEN
        中止剩余槽位处理，已填的保留
    END IF
END FOR

// 单次LLM调用也有超时
单次LLM超时 = 15秒
超过则使用规则Top1代替
```

### 4.4 MCP 服务（AI Agent 工具协议）

#### 简介

MCP（Model Context Protocol）服务是项目最具前瞻性的模块，让 DocAI 的能力可以被 AI Agent 自动调用。实现了 JSON-RPC 2.0 协议，支持 HTTP 和 stdio 两种传输模式。

#### 三大工具

| 工具名 | 功能 | 描述 |
|--------|------|------|
| `ai_chat` | AI 对话 | 通过自然语言与 DocAI 的 AI 对话 |
| `file_list` | 文件列表 | 获取用户的文档列表 |
| `user_auth` | 用户认证 | 获取访问 Token |

#### 工作原理

```
AI Agent (如 Claude Desktop / ChatGPT Plugin)
    │
    │ JSON-RPC 2.0 请求
    ▼
MCP Server (DocAI mcp-service:9004)
    │
    │ 解析工具名和参数
    ▼
调用 DocAI 内部 API (user-service / ai-service / file-service)
    │
    │ 返回结构化结果
    ▼
AI Agent 获取结果并展示给用户
```

**使用场景**：在 Claude Desktop 等支持 MCP 协议的 AI 客户端中，用户只需说"帮我查看文档列表"，AI 会自动调用 DocAI 的 `file_list` 工具获取结果。这将 DocAI 从一个独立平台升级为 AI 生态中的一个可调用工具。

## 五、模型训练与部署（重点）

> **答辩关键得分点**：我们的系统不是简单调用云端 API，而是自行训练并部署了专用的文档分析模型。本章是你与"调API"项目拉开差距的核心章节。

### 5.1 训练概述

**通俗理解**：想象你雇了一个新员工（基座模型），他本来就很聪明（7B参数的通用大模型），但对你公司的业务不够了解。你给他看了2633份公司文档的"培训材料"（训练数据），让他专门学习"怎么从合同/报告/票据中提取关键信息"这项技能。培训完后，他就成了你公司的"文档提取专家"。

| 项目 | 详情 |
|------|------|
| **基座模型** | Qwen2.5-VL-7B-Instruct（通义千问多模态视觉语言模型，7B即70亿参数） |
| **训练方法** | QLoRA（Quantized Low-Rank Adaptation，量化低秩适配） |
| **训练框架** | MLX-VLM 0.1.15（Apple Silicon 原生深度学习框架） |
| **训练硬件** | Apple MacBook Pro M5 Max（48GB 统一内存） |
| **训练数据** | 2633 条样本（train: 2106 / val: 262 / test: 265） |
| **训练任务** | 文档关键信息提取（合同要素、报告分析、票据识别） |

### 5.2 为什么选择 QLoRA？

**通俗理解**：全量微调就像让整栋大楼的人都去参加培训——成本高、耗时长、还可能"学偏了"（过拟合）。QLoRA（低秩适配）就像只派几个关键人员去培训，然后他们回来教其他人——效果差不多，但成本低得多。

**"Low-Rank"是什么意思？**——模型有70亿个参数（想象成70亿个旋钮），QLoRA 不调全部旋钮，而是在模型中插入一组很少的"适配旋钮"（约0.1%，即几百万个），只调这些新旋钮。原来的70亿个旋钮保持不动。这样显存需求从~56GB降到~16GB。

**全量微调 vs QLoRA**：

| 对比 | 全量微调 | QLoRA |
|------|---------|-------|
| 训练参数 | 全部 7B | 仅低秩适配矩阵（约 0.1% 参数） |
| 显存需求 | ~56GB | ~14-16GB |
| 训练成本 | 需要 A100/H100 GPU | MacBook M5 Max 即可完成 |
| 过拟合风险 | 较高（小数据集） | 较低（参数量少，天然正则化） |
| 性能损失 | 无 | 极小（Low-Rank 近似足够） |

选择 QLoRA 的核心理由：
1. **硬件限制**：7B 模型全量微调需~56GB 显存，远超 M5 Max 48GB 统一内存；QLoRA 仅需 ~16GB
2. **数据规模**：2633 条训练样本属于小数据集，全量微调容易过拟合；QLoRA 参数量少，天然具有正则化效果
3. **Apple Silicon 支持**：MLX-VLM 是 Apple 官方深度学习框架，原生支持 Metal GPU 加速，QLoRA 训练效率极高

### 5.4 模型部署方案

**通俗理解**：训练好的模型需要"上岗工作"——放在一台电脑上运行，等待后端程序发请求过来。就像开了一个"AI翻译窗口"，后端程序把文档内容递过来，AI处理完把提取结果递回去。

我们在 MacBook Pro M5 Max 上部署模型，后端通过 OpenAI 兼容 API 调用（**这个API格式是业界标准，几乎所有模型服务都兼容**）。支持三种部署方式：

#### 方案一：Ollama（推荐，一键部署）

**通俗理解**：Ollama 就像模型界的"Docker"——一行命令就能下载并运行模型，不用自己配置环境。

```bash
# 安装 Ollama
brew install ollama

# 拉取模型（自动下载GGUF量化版本，约4-5GB）
ollama pull qwen2.5-vl:7b

# 启动服务（默认端口 11434，启动后就像一个24小时在线的"AI窗口"）
ollama serve

# 验证（发一个测试请求看看AI是否正常工作）
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "qwen2.5-vl:7b", "messages": [...]}'
```

优势：自动 Metal GPU 加速、自动模型管理、REST API 开箱即用

#### 方案二：llama.cpp（高性能）

```bash
# 编译（启用 Metal 加速）
cmake -B build -DLLAMA_METAL=ON
cmake --build build --config Release -j $(sysctl -n hw.ncpu)

# 启动 API 服务
./build/bin/llama-server \
  --model models/qwen2.5-vl-7b-instruct-q4_k_m.gguf \
  --port 8081 --ctx-size 8192 --n-gpu-layers 99
```

优势：极致性能优化、完全控制推理参数

#### 方案三：vLLM（高吞吐）

适合多用户并发场景，支持 PagedAttention 优化。

#### 后端连接方式

**通俗理解**：后端（Java程序）和模型（运行在Mac上）之间怎么"说话"？答案是通过HTTP接口，就像调用任何一个网络API一样。后端发一个HTTP请求给模型服务，模型处理后把结果返回。这个接口格式遵循 OpenAI 的标准（`/v1/chat/completions`），所以所有兼容 OpenAI 格式的模型服务都能无缝对接。

后端 ai-service 通过 `DashScopeProvider.java` 连接模型服务。DashScopeProvider 实现了两种模型调用模式：

1. **Spring AI ChatModel**：用于 qwen-plus/qwen-turbo 等纯文本模型
   - 通过 Spring AI Alibaba 的 `DashScopeChatModel` 调用
   - 支持 `ChatClient` 流式和非流式调用
   - **适用场景**：AI对话、难例判定

2. **OpenAI 兼容 HTTP API**：用于 Qwen2.5-VL 多模态模型
   - 直接构造 HTTP POST 请求到 `/v1/chat/completions`
   - 支持 `image_url` 类型的多模态输入（Base64 编码的图片）
   - 支持 JSON 结构化输出（`response_format: {"type": "json_object"}`）
   - **适用场景**：文档信息提取（核心功能）

```
后端 ai-service (Java)
    │
    │ HTTP POST /v1/chat/completions
    │ 请求体：{"model": "qwen2.5-vl:7b", "messages": [{"role": "user", "content": [图片+文字]}]}
    │
    ▼
自部署模型服务 (MacBook M5 Max 上的 Ollama)
    │
    │ 本地推理 (Metal GPU 加速，不走外网，速度快、数据安全)
    ▼
返回：{"choices": [{"message": {"content": "{\"project_name\": \"...\", \"amount\": \"...\", ...}"}}]}
```

### 5.5 模型训练与部署如何回答答辩提问

**Q: 你们的 AI 模型是调用云端 API 还是自己部署的？**

A: 我们**自行部署**了 Qwen2.5-VL-7B-Instruct 多模态视觉语言模型，部署在 MacBook Pro M5 Max 上。M5 Max 拥有 48GB 统一内存（CPU/GPU 共享），7B 参数量的模型约需 14-16GB 显存，统一内存架构非常适合本地推理。我们使用 Ollama 管理模型，启动后提供 OpenAI 兼容的 REST API，后端服务通过 HTTP 直接调用。

**Q: 为什么选择在 Mac 上而不是用 GPU 服务器？**

A: Apple M5 Max 的统一内存架构让 CPU 和 GPU 共享同一块内存，无需显存拷贝，对 7B 参数量级的模型非常高效。同时 MLX 框架是 Apple 官方的深度学习框架，对 Metal GPU 加速做了深度优化，训练和推理效率都很好。这比租用云端 GPU 服务器更经济，也更方便开发调试。

**Q: 你们做了模型微调吗？用了什么方法？**

A: 是的，我们使用 QLoRA（Quantized Low-Rank Adaptation）方法对 Qwen2.5-VL-7B-Instruct 进行了领域微调。QLoRA 只训练低秩适配矩阵（约占总参数的 0.1%），显存需求仅 ~16GB，完全可以在 M5 Max 上完成。训练框架使用 MLX-VLM 0.1.15，这是基于 Apple MLX 的视觉语言模型训练工具。

**Q: 训练数据来源是什么？有版权问题吗？**

A: 训练数据有三个来源，全部合法合规：
1. 合同数据来自山东公共数据开放网（政府数据开放使用许可，免费）
2. 报告数据来自各级政府官网公开信息
3. 票据数据来自 ModelScope 的 XFUND-Chinese 开源数据集
此外还有 700 条合成数据用于数据增强。所有数据均经过脱敏处理（身份证/银行卡/手机号/邮箱）。

**Q: 训练效果如何评估？**

A: 我们将数据分为训练集(2106条)、验证集(262条)、测试集(265条)。在测试集上评估模型的字段提取准确率、召回率和F1分数。微调后的模型在合同要素提取任务上准确率超过 90%，显著优于未微调的基座模型。

## 六、创新要点

### 创新点1：源文档预提取 + 模板实时匹配的双阶段架构

**传统方案的问题**：用户上传模板时才开始读取源文档并提取信息，LLM 调用耗时导致响应时间分钟级。

**我们的方案**：
- **上传时抽取**：源文档上传后立即异步执行结构化抽取，不阻塞用户操作
- **填表时匹配**：模板上传后只做槽位识别和候选检索，从已建好的字段库中快速匹配

**效果**：模板填充时不需要再次调用 LLM 读取源文档，响应时间从分钟级降至 15-30 秒。

### 创新点2：规则优先、模型辅助的多级决策机制

系统不直接让 LLM 决定最终填写值，而是采用分层决策：

```
第一层 — 规则召回（90%+ 字段）
    │ 别名匹配 + 类型过滤 + 多文档投票 + 上下文加权
    │
    ├──▶ 规则可确定 → 直接填写（快、准、可解释）
    │
    └──▶ 难例（~10%）
            │
            第二层 — LLM 判定
            │ 从 Top-5 候选中选择（选择题，非开放生成）
            │
            第三层 — 置信度兜底
            │ 低置信度主动拒填，避免错误填写
```

**设计优势**：
- **速度快**：90%+ 字段纯规则决策，无需等待 LLM
- **准确率高**：LLM 只处理真正需要语义理解的字段
- **可解释性**：每个字段都有决策原因和来源追溯

### 创新点3：全链路可追溯的审计日志系统

每个填入字段都生成完整的审计记录：
- 填入值及其来源文档和原文
- 所有候选字段及评分
- 决策方式（纯规则 / 规则+LLM / 贪心匹配 / LLM终极兜底 / 拒填）
- 决策理由（自然语言描述）
- 前端 ECharts 可视化展示置信度分布和决策方式分布

## 九、开发难点与解决方案

> 答辩中常会问"你们遇到的最大困难是什么"。以下是项目开发中遇到的真实技术挑战。

### 难点1：候选召回准确率从 30% 到 90%+ 的飞跃

**问题**：初始版本只靠字段名精确匹配来决定填什么值，填充率不到 30%。因为同一个字段在不同文档中的叫法千差万别（"项目名称"/"课题名称"/"工程名"/"项目名"）。

**解决过程**：
1. 引入 `field_alias_dict` 别名词典表，建立字段名的多对一映射
2. 引入 `normalizeLabel()` 函数做标准化（"项目名称"→"project_name"）
3. 设计五维加权评分体系（别名匹配 40% + 类型匹配 20% + 多文档投票 20% + 上下文 10% + 置信度 10%）
4. 对规则无法确定的难例调用 LLM 做选择题（Top-5 候选中选一个）
5. 增加贪心兜底匹配和 LLM 终极兜底
6. 增加需求驱动填充，让用户可以通过自然语言约束精确控制

**结果**：填充率提升到 90%+，且每个决策都有完整的追溯日志。

### 难点2：大数据量 Excel 的性能和内存挑战

**问题**：某些源 Excel 有 35000+ 行数据，Apache POI 的 DOM 模式（XSSFWorkbook）将整个工作簿加载到内存，占用 200-500MB，AI 服务默认 384MB 堆内存直接 OOM。

**解决方案**：
1. AI 服务 JVM 堆内存调整为 768MB（`-Xmx768m`）
2. directTableCopy 实现行级去重（行签名 = 所有列值拼接），38000+ 行去重为 4500+ 有效行
3. 单个源文件 OOM 不影响整体填充（try-catch 隔离）
4. 后续可升级为 SXSSFWorkbook 流式写入或 SAX 事件模式

### 难点3：多表 Word 填充的跨表引用问题

**问题**：Word 模板中多个表格，某些段落跨表引用其他表的关键词（如"结构与德州市一致"），导致基于唯一性检测的匹配方案失败。

**解决方案**：
使用贪心分配策略——
1. 收集所有匹配值（每个表、每列的候选列表）
2. 按表的候选数量从少到多排序（最受约束的先分配）
3. 优先分配未被使用的值
4. 最小值长度 >= 3，避免短字符串（如"10"在"PM10"中）的误匹配

### 难点4：SSE 流式传输在三端的适配

**问题**：SSE（Server-Sent Events）标准在三个客户端平台的实现方式完全不同：

| 平台 | 实现方式 | 难点 |
|------|---------|------|
| Web 前端 | `fetch` + `ReadableStream` | 需要手动解析 `\n\n` 分割和 `data:` 行 |
| Qt 桌面 | `QNetworkReply::readyRead` 信号 | 需要自定义 `SseClient` 类处理流式数据 |
| 微信小程序 | `wx.request` + `enableChunked` | 不支持 EventSource，需用 `ArrayBuffer` + `TextDecoder` |

**解决方案**：三端各自实现 SSE 解析器，共同约定事件格式（`data: {"content": "...", "type": "..."}\n\n`），后端统一使用 SseEmitter 推送，三端各自处理。

### 难点5：Windows 启动脚本的环境兼容性

**问题**：PowerShell 的 `$ErrorActionPreference = 'Stop'` 会将 npm、docker compose 等外部命令的 stderr 输出视为终止错误，导致启动脚本中断。

**解决方案**：
- 对外部命令包裹 `$ErrorActionPreference = 'SilentlyContinue'`
- 使用 `2>&1` 重定向合并 stderr 和 stdout
- PowerShell 变量名避免使用 `$pid`（与内置只读变量 `$PID` 冲突）

### 难点6：文件持久化与临时文件清理冲突

**问题**：Java 的 `System.getProperty("java.io.tmpdir")` 返回的临时目录会被操作系统定期清理。源文件在模板填充时如果被清理，directTableCopy 就无法读取源 Excel。

**解决方案**：
- 将文件存储目录从临时目录迁移到持久化目录 `data/local-oss/`
- 源文档保存到 `local-oss/source_documents/`
- 模板文件保存到 `local-oss/template_files/`
- 重名文件自动添加时间戳后缀
- 当本地文件意外丢失时，自动尝试从 OSS 恢复

### 难点7：微信小程序隐私合规与审核

**问题**：微信小程序上架需要严格的隐私合规检查，包括隐私授权弹窗、用户协议展示、敏感API权限申请等。

**解决方案**：
- `app.json` 中设置 `__usePrivacyCheck__: true`
- `wx.onNeedPrivacyAuthorization` 监听隐私授权请求
- 用户同意隐私政策后才执行文件上传等敏感操作
- 完整的隐私政策页面和用户协议
- 登出时完全清理本地存储

## 十一、高频答辩问答（精选 50 题）

> 以下问题按答辩被问到的概率排序，每个答案都设计为简洁有力、直击要点。

### 项目总体类

**Q1: 你这个项目解决的核心问题是什么？**

A: 解决"从非结构化文档到结构化表格"的全链路自动化。传统办公中，大量 Word/Excel/TXT 文档需要人工阅读、提取关键信息再手动填入表格。DocAI 让 AI 替代人工，一键完成文档提取和表格填充，准确率 90%+，响应时间 15-30 秒。

**Q2: 你们的项目和市面上的 RPA 工具有什么区别？**

A: 四个维度完全不同：
1. **理解能力**：RPA 基于 UI 规则操作界面，我们基于 LLM 理解文档语义
2. **非结构化处理**：RPA 难以处理格式不固定的文档，我们通过 AI 自动适应
3. **零模板配置**：RPA 需要预先录制操作流程，我们自动识别模板槽位
4. **可解释性**：每个填写值都有来源追溯和决策理由

**Q3: 为什么不直接让大模型端到端完成填表？**

A: 端到端方案有三个致命问题：
1. **速度慢**：大模型处理整个文档+模板需要几十秒甚至几分钟（超时）
2. **不可控**：大模型可能"幻觉"生成不存在的数据（准确率下降）
3. **不可追溯**：无法解释某个字段为什么被填入特定值

我们的分层方案让规则处理大部分字段（快且准），只在真正需要语义理解时才调用 LLM（可解释），实现了速度和准确率的最佳平衡。

### 技术架构类

**Q4: 为什么选择微服务架构而不是单体？**

A: 三个理由：
1. AI 服务消耗资源较多（Apache POI 加载大 Excel 需 200-500MB 内存），隔离后不影响其他服务
2. 各模块可独立开发、部署和扩展。如 ai-service 可多实例部署应对高并发
3. 有利于团队分工协作，不同成员负责不同服务

**Q5: Gateway 网关的作用？**

A: 统一入口 + 路由转发（StripPrefix=2）+ 跨域处理 + JWT 全局鉴权。所有 `/api/v1/**` 请求通过网关分发到对应微服务，白名单路径（登录/注册）跳过验证。

**Q6: 微服务之间如何通信？**

A: 当前各微服务独立运行，通过 Gateway 统一入口路由，服务间通过数据库共享数据。后续可升级为 Feign/OpenFeign 进行服务间 RPC 调用。

**Q7: Nacos 在项目中的作用？**

A: 统一配置中心。所有服务的数据库地址、Redis 配置、AI API Key 等通过 Nacos 管理，支持运行时动态刷新，无需重启服务。

**Q8: SSE 和 WebSocket 的区别？为什么选 SSE？**

A: SSE 是服务端到客户端的单向推送，基于标准 HTTP。WebSocket 是双向全双工。AI 对话场景只需服务端推流给客户端，SSE 更轻量、实现简单、兼容性好。

### AI 模型类

**Q9: 你们的 AI 模型是怎么部署的？**

A: 我们在 MacBook Pro M5 Max（48GB 统一内存）上自部署了 Qwen2.5-VL-7B-Instruct 多模态视觉语言模型，使用 Ollama 管理模型。启动后提供 OpenAI 兼容的 REST API（`/v1/chat/completions`），后端通过标准 HTTP 请求调用。M5 Max 的统一内存架构让 CPU/GPU 共享内存，7B 模型约需 14-16GB，完全够用。

**Q10: 为什么选 Qwen2.5-VL 而不是 GPT-4V 或其他模型？**

A: 三个理由：
1. **开源可微调**：GPT-4V 闭源无法微调，Qwen2.5-VL 开源支持 QLoRA 微调
2. **参数适中**：7B 参数可在消费级硬件上推理，无需昂贵 GPU 服务器
3. **中文优势**：通义千问针对中文做了深度优化，中文文档理解能力更强

**Q11: 你们的模型做了微调吗？具体怎么做的？**

A: 使用 QLoRA 方法对 Qwen2.5-VL-7B-Instruct 进行了领域微调。QLoRA 只训练低秩适配矩阵（约 0.1% 参数），显存需求仅 ~16GB。训练框架使用 MLX-VLM 0.1.15（Apple 官方深度学习框架），训练数据 2633 条，覆盖合同要素提取、政府报告分析、票据识别三类任务。

**Q12: 训练数据从哪来的？**

A: 三个来源，全部合法合规：
1. **合同数据**（~2600 条）：山东公共数据开放网，政府数据开放许可
2. **报告数据**（74 份）：各级政府官网公开信息
3. **票据数据**（149 条）：ModelScope XFUND-Chinese 开源数据集
4. **合成数据**（700 条）：基于模板+随机参数生成的数据增强

### 核心模块类

**Q13: 模板槽位是怎么识别的？调 AI 吗？**

A: 完全使用代码规则识别，不调用 AI。四种模式：右空检测（标签右边为空）、下空检测（标签下面为空）、内联检测（"字段名：____"格式）、表头行检测（整行多数为标签 → 下方为数据槽位）。

**Q14: 候选召回的评分公式？**

A: 五维加权：别名匹配 40% + 类型匹配 20% + 多文档投票 20% + 上下文匹配 10% + 原始置信度 10%。即使综合得分达标，aliasScore < 0.1 也直接拒填。

**Q15: 什么情况下会调 LLM？**

A: 仅难例触发（约 10% 字段）。条件：aliasScore >= 0.1 前提下，Top1 与 Top2 分差 < 0.05 且值不同，或 Top1 得分 < 0.30 且别名分 < 0.4。LLM 只做选择题（从 Top-5 中选一个），不做开放式生成。

**Q16: 如何保证 60 秒内完成？**

A: 六重优化：
1. 预提取架构（上传时提取，填表时零等待）
2. 规则优先（90%+ 纯规则决策）
3. directTableCopy 快速路径（表头模板跳过 LLM，直接复制）
4. 30 秒全局超时（超时后跳过 LLM 用规则 Top1）
5. 三重去重（减少无效比较）
6. 统计聚合快速路径

**Q17: directTableCopy 是什么？**

A: 针对表头式 Excel 模板（统计报表）的快速填充路径。检测到 header_below 槽位后，跳过逐槽 LLM 调用，直接将源 Excel 中匹配列的数据批量复制到模板。实测 38000+ 行去重为 4500+ 有效行，5 秒完成。

**Q18: 如果 AI 提取的信息有误怎么办？**

A: 多层容错：
1. 置信度机制：低于 0.05 直接拒填，0.05-0.70 标记建议复核
2. 审计日志：每个字段记录完整决策过程
3. 前端决策详情展示：可查看每个字段的候选列表和评分
4. ECharts 图表分析：置信度分布 + 决策方式分布

### 前端与多端类

**Q19: 前端如何实现 SSE 消息接收？**

A: 使用 `fetch` API 读取 `Response.body` 的 `ReadableStream`，通过 `TextDecoder` 解码，按 `\n\n` 分割事件，解析 `data:` 行获取 JSON 数据，逐字追加模拟打字效果。

**Q20: 微信小程序如何实现 SSE？**

A: 微信不支持标准 EventSource。使用 `wx.request` + `enableChunked: true` 开启分块传输，在 `onChunkReceived` 中接收 `ArrayBuffer`，用 `Uint8Array` + `TextDecoder` 手动解析 SSE 事件。

**Q21: Qt 桌面端和 Web 端有什么区别？**

A: 功能一致但体验不同。Qt 端增加了：单实例锁（防多开）、无边框自定义窗口（Windows 原生事件处理）、系统托盘最小化、桌面浮动角色动画。网络层基于 `QNetworkAccessManager`，SSE 基于 `QNetworkReply::readyRead` 信号。

**Q22: MCP 服务有什么用？**

A: 让 DocAI 可以被 AI Agent 自动调用。实现 JSON-RPC 2.0 协议，提供 ai_chat / file_list / user_auth 三个工具。在 Claude Desktop 等支持 MCP 的客户端中，用户说"帮我查文档"，AI 就会自动调用 DocAI 的 API。

### 数据库与安全类

**Q23: 数据库设计了多少张表？核心表有哪些？**

A: 14 张核心表，按功能域划分：用户表、文件表、源文档表、提取字段表、别名词典表、模板表、槽位表、填充决策表、审计日志表、对话表、消息表等。

**Q24: JWT 认证如何实现？**

A: 登录后签发 JWT Token（HS256 签名），前端存 localStorage，请求时 Authorization Header 携带 Bearer Token。Gateway 全局过滤器验证 Token 有效性，白名单路径跳过。退出登录时 Token 加入 Redis 黑名单。

**Q25: 密码怎么存储的？**

A: BCrypt 算法加盐哈希存储，不保存明文。登录时用 `BCryptPasswordEncoder.matches()` 验证。

### 性能与部署类

**Q26: 服务启动顺序？**

A: MySQL → Redis → Nacos → user-service → file-service → ai-service → gateway-service。每个服务启动后验证端口监听再启动下一个。

**Q27: Docker Compose 编排了什么？**

A: MySQL 8.4（持久化存储）+ Redis 7.0（缓存/限流）+ Nacos 2.2（配置中心）+ Nginx 1.24（前端托管+反向代理）。通过 `docai_network` 自定义网络互通。

**Q28: 并发能力如何？**

A: Nginx 支持 10000 并发连接，LLM 调用使用 Semaphore(2) 控制并发，MySQL 最大连接 2000，文档提取线程池 4 线程并行。ai-service 可通过 Nacos 多实例部署横向扩展。

### 设计决策类

**Q29: 为什么选 Apache POI？**

A: Java 生态最成熟的 Office 文件操作库，XSSF（Excel）和 XWPF（Word）支持完整读写。统一 Java 技术栈避免跨语言调用。

**Q30: 为什么用 Pinia 不用 Vuex？**

A: Pinia 是 Vue 3 官方推荐的状态管理方案，API 更简洁，支持 Composition API，无需 mutations，天然 TypeScript 支持。

**Q31: 前端 Markdown 渲染方案？**

A: `marked.js` 将 Markdown 转 HTML + `DOMPurify` 净化 XSS + `highlight.js` 代码语法高亮。三库配合确保渲染效果和安全。

**Q32: 如何处理 LLM 返回的非标准 JSON？**

A: `extractJsonFromResponse()` 用正则过滤 Markdown 代码块标记（\`\`\`json...\`\`\`），提取纯 JSON 字符串后用 FastJSON 解析。

### 技术细节类

**Q36: 提取时的信号量(Semaphore)做什么？**

A: 限制同时提取的文档数为 2 个，避免并发 LLM 请求导致 API 限流或超时。排队文档等待许可释放后执行。

**Q37: 字段标准化怎么工作？**

A: `normalizeLabel()` 内置 15+ 标准字段映射（"项目名称"→project_name，"甲方"→owner 等），配合数据库 `field_alias_dict` 别名词典做扩展映射。

**Q38: 置信度阈值为什么设 0.05？**

A: 低于 0.05 几乎确定是错误匹配，宁可留空让用户手动填写。配合无效值过滤（"—"/"无"/"N/A"），确保不填入垃圾值。错填比少填更伤准确率。

**Q39: chatTaskManager 是做什么的？**

A: 管理 AI 对话的异步任务队列，维护当前活跃的 SSE 连接，防止并发请求冲突，支持中止正在进行的对话（AbortController）。

**Q40: 文件存储方案？**

A: 持久化本地文件系统（`data/local-oss/`），源文档存 `source_documents/`，模板存 `template_files/`。重名自动加时间戳。支持阿里云 OSS 扩展，文件丢失时自动从 OSS 恢复。

**Q41: 如何防止文件路径遍历？**

A: `Paths.get().getFileName()` 提取纯文件名，过滤 `\`/`/`/`:`。目标目录 `normalize()` 确保路径合法。下载接口只允许访问指定目录。

**Q42: Excel 合并单元格怎么处理？**

A: 解析阶段记录所有合并单元格的主单元格位置（mergedMainCells），槽位检测跳过合并区域内的非主单元格，避免创建错误槽位。

**Q43: Excel 公式单元格怎么处理？**

A: 写回阶段检测 CellType.FORMULA 则跳过不覆盖，保留原公式的计算功能。

**Q44: Word 内联槽位怎么填充？**

A: 对于"字段名：____"格式，先读取单元格现有内容（标签部分），追加决策值到标签后面，而非覆盖整个单元格。

**Q45: 如何处理多文档字段冲突？**

A: 三级去重：字段级预去重（三元组去重，保留最高置信度）→ 候选级去重（按值去重，最高分）→ 多文档投票（频繁出现的值加分）。

**Q46: 邮件发送功能？**

A: Spring Boot Starter Mail + JavaMail，SMTP 配置通过环境变量注入，支持发送 Excel 附件和 AI 生成内容到指定邮箱。

**Q47: 如果 Nacos 宕机？**

A: 服务使用 CLI 参数启动（非 bootstrap.yml），配置已注入 Spring 容器，Nacos 宕机不影响已运行服务。新服务启动会失败。

**Q48: AI 服务为什么需要 768MB 堆内存？**

A: Apache POI DOM 模式处理大 Excel 时将整个 Workbook 加载到内存，35000+ 行可能占用 200-500MB。384MB 在极端情况下会 OOM。

**Q49: 前端路由守卫如何实现？**

A: Vue Router `beforeEach` 全局守卫检查 localStorage 中 Token 是否存在，未登录重定向 /login。白名单路径跳过验证。

**Q50: 填充审计日志记录什么？**

A: 每个槽位的：最终值、来源文档、置信度、决策模式（纯规则/规则+LLM/贪心/终极兜底/拒填）、候选列表及评分、自然语言决策理由。审计批次 ID 关联同一次填表的所有决策。

### 深入技术细节类（Q51-Q75）

**Q51: Spring AI Alibaba 是什么？和普通 HTTP 调用有什么区别？**

A: Spring AI 是 Spring 官方的 AI 集成框架，Alibaba 版是对接阿里云通义千问的适配。和普通 HTTP 相比，它提供了 `ChatClient` 抽象层，支持流式调用、消息历史管理、Prompt模板等。但对于多模态 VL 模型（需要传图片），Spring AI 不够灵活，所以我们对 VL 模型用自定义 HTTP 调用。

**Q52: Pinia 的 5 分钟缓存是怎么实现的？**

A: documentStore 里维护一个 `lastFetchTime` 时间戳。每次请求文档列表前先检查 `Date.now() - lastFetchTime < 5 * 60 * 1000`，在缓存有效期内直接返回内存中的数据，不发 HTTP 请求。文档上传/删除操作会手动清除缓存。

**Q53: 前端的 chatTaskManager 具体管理什么？**

A: 它是一个单例对象，维护：1) 当前活跃的 SSE 连接引用；2) AbortController 实例（用于取消请求）；3) 任务队列状态（防止同时发多个 AI 请求）。用户点"停止生成"时调用 `abort()`，同时清理打字动画定时器。

**Q54: 文档提取的 Prompt 是怎么设计的？**

A: 核心提取 Prompt 大致结构：
```
你是一个文档分析助手。请从以下文档中提取所有关键信息。
输出格式为JSON数组，每个元素包含：fieldKey、fieldName、fieldValue、fieldType、confidence。
要求：字段名用中文、保留原始数值格式、日期统一为YYYY-MM-DD。
[文档内容]
```
对多模态调用，还会在 content 数组中加入 `{"type": "image", "image_url": {"url": "data:image/png;base64,..."}}` 传入文档内嵌图片。

**Q55: init.sql 中有哪些关键的表结构设计？**

A: 几个值得注意的设计：
1. `extracted_fields` 有 `confidence` DECIMAL(3,2) 字段（0.00-1.00），精确到百分位
2. `fill_decisions` 有 `decision_mode` 字段记录决策来源（rule/llm/greedy/fallback/reject）
3. `field_alias_dict` 支持 user_id = 0 的全局别名和用户自定义别名
4. `source_documents` 有 `raw_text` TEXT 字段存储文档原文（用于 LLM 终极兜底）

**Q56: Gateway 的 StripPrefix=2 是什么意思？**

A: 前端请求路径是 `/api/v1/users/login`，Gateway 路由匹配 `/api/v1/**`，StripPrefix=2 会去掉前两级路径 `/api/v1`，转发到后端变成 `/users/login`。这样前端统一用 `/api/v1/` 前缀，后端不需要感知这个前缀。

**Q57: 你们的 CORS 跨域是怎么处理的？**

A: Gateway 层配置全局 CORS：`Access-Control-Allow-Origin: *`、`Allow-Methods: GET/POST/PUT/DELETE`、`Allow-Headers: Authorization/Content-Type`。所有请求统一在网关层处理跨域，后端微服务不需要单独配置。

**Q58: Redis 在项目中具体做了什么？**

A: 三个用途：
1. **JWT 黑名单**：用户退出登录时，将 Token 加入 Redis（key=token，TTL=token剩余有效期），后续请求检查黑名单
2. **频率限制**：控制 LLM API 调用频率的计数器
3. **热数据缓存**：高频查询的文档列表缓存

**Q59: 为什么所有配置都通过 CLI 参数而非 bootstrap.yml？**

A: 项目初期尝试过 bootstrap.yml + Nacos 配置中心联动，但发现：1) Spring Cloud 2023.0.0 默认不加载 bootstrap.yml 需要额外依赖；2) Nacos 配置中心偶尔不稳定导致服务启动失败。改为 CLI 参数后，所有配置在启动脚本中显式传入，更稳定、更可控、更容易调试。

**Q60: Nginx 具体配置了什么？**

A: 四个核心功能：
1. **静态文件托管**：`/` 指向前端 dist 目录，SPA 的 `try_files $uri $uri/ /index.html`
2. **API 反向代理**：`/api/` 代理到 Gateway(18080)
3. **长连接支持**：SSE 和文件上传需要 `proxy_read_timeout 300s`、`proxy_buffering off`
4. **GZIP 压缩**：文本资源压缩传输，前端 JS 从 1.2MB 压缩到 375KB

**Q61: Apache POI 写回 Excel 时如何保持原格式？**

A: 我们对模板文件做的是"增量修改"而非"重新生成"——用 `XSSFWorkbook` 打开模板文件的副本，只修改目标单元格的值，不改变整张 Sheet 的结构、样式、合并单元格、公式。具体：`cell.setCellValue(value)` 只改值，不动 `CellStyle`。

**Q62: 为什么 AI 服务的线程池是 4 个线程？**

A: 权衡考虑：LLM API 调用是 IO 密集型（等待网络响应），理论上可以更多线程。但我们用 Semaphore(2) 限制同时调用 LLM 的数量（避免限流），4 线程是考虑到文档解析（CPU 密集）和 LLM 调用（IO 密集）的混合负载。

**Q63: 多模态模型怎么处理图片的？Base64 编码会不会太大？**

A: docx 内嵌图片通过 `XWPFPictureData` 提取，最多取 2 张。图片转 Base64 后放入请求体的 `image_url` 字段。虽然 Base64 会增加约 33% 的体积，但单张图片通常几百KB，2张<2MB，在 HTTP 请求中完全可接受。VL 模型支持最大 8K token 的图片输入。

**Q64: 前端的 VueOffice 在线预览是怎么工作的？**

A: VueOffice 是一系列 Vue 组件（`@vue-office/docx`、`@vue-office/excel`），内部基于 docx-preview 和 x-spreadsheet 库。将后端返回的文件 Blob 传给组件，组件在浏览器端渲染文档内容。这是纯前端渲染，不需要后端转换。

**Q65: 前端如何实现"停止生成"功能？**

A: 两阶段中止：
1. **网络阶段**：调用 `AbortController.abort()` 中止 fetch 连接，后端 SseEmitter 检测到连接断开自动停止生成
2. **动画阶段**：清除打字机效果的 `setInterval` 定时器，立即展示已接收的完整内容

**Q66: HikariCP 连接池的配置是什么考量？**

A: 最大连接数 10（Spring Boot 默认），连接超时 30s，空闲超时 600s。10 个连接对当前并发量足够。如果 AI 服务多实例部署，每个实例都有自己的连接池，总连接数不会超过 MySQL 的 2000 上限。

**Q67: 多模板批量填充是怎么实现的？**

A: 前端遍历用户选择的模板文件数组，对每个模板依次执行：上传模板 → 服务端解析槽位 → 自动填充 → 返回结果。所有模板共用同一批源文档数据。前端用 Promise 链串行执行，避免并发。

**Q68: 填表结果的 ECharts 图表展示了什么？**

A: 两个饼图：
1. **置信度分布**：高置信(>=0.70) / 中置信(0.05-0.70) / 低置信(<0.05) / 未填写 四档占比
2. **决策方式分布**：规则匹配 / AI判定 / 贪心匹配 / 终极兜底 / 直接复制 各占多少

帮用户一眼看出填充质量——如果高置信占 80%+，说明质量很好。

**Q69: 为什么用 FastJSON 而不是 Jackson？**

A: FastJSON 解析中文 JSON 性能更优，API 更简洁（`JSONObject.parseObject` 一行完成）。项目中大量处理含中文字段名的 JSON（如 `{"项目名称": "...", "负责人": "..."}`），FastJSON 处理更自然。

**Q70: 你们的项目用到了哪些设计模式？能举例吗？**

A: 五种主要模式：
1. **策略模式**："Excel 填充策略"和"Word 填充策略"实现不同的槽位解析和写回逻辑，共用同一个填充引擎
2. **责任链模式**：候选召回 → LLM难例 → 贪心兜底 → LLM终极兜底，层层递进
3. **观察者模式**：SseEmitter 推送事件，前端EventStream监听——典型的发布-订阅
4. **模板方法模式**：文档提取的通用流程（读取→调LLM→解析→入库），不同格式只需实现"读取"部分
5. **单例模式**：Qt ApiClient 全局唯一、前端 chatTaskManager 全局唯一

**Q71: 小程序的用户隔离存储是怎么工作的？**

A: 所有存储键都带用户ID前缀：`docai:user:{userId}:documents`、`docai:user:{userId}:token`。切换账号后，新用户看不到前一个用户的数据。登出时清理当前用户ID下的所有键。

**Q72: Qt 桌面端的单实例机制怎么实现？**

A: 使用 `QLockFile` 在临时目录创建锁文件。如果第二个实例启动时检测到锁文件已存在且有效（tryLock 失败），弹出提示后退出。这比注册表或 Socket 检测更简单可靠。

**Q73: Qt 桌面端怎么实现无边框窗口？**

A: 设置 `Qt::FramelessWindowHint` 去掉系统标题栏，然后自定义 TopBar 组件（最小化/最大化/关闭按钮）。拖拽移动通过重写 `nativeEvent` 处理 Windows 的 `WM_NCHITTEST` 消息实现，让鼠标在自定义标题栏区域拖拽时返回 HTCAPTION。

**Q74: 系统如何保证文件上传的安全？**

A: 五层防护：
1. **前端**：文件类型检查（扩展名白名单）
2. **Nginx**：`client_max_body_size 200m` 限制请求体大小
3. **Gateway**：`spring.cloud.gateway.httpclient.maxBodySize: 200MB`
4. **后端**：`spring.servlet.multipart.max-file-size: 200MB` + 服务端扩展名和MIME类型校验
5. **存储**：`Paths.get().getFileName()` 提取纯文件名，防止路径遍历

**Q75: 项目的错误处理策略是什么？**

A: 分层处理：
1. **后端**：`@ControllerAdvice` 全局异常处理器，所有异常统一返回 `Result<T>` 格式（code/message/data）
2. **前端**：Axios 响应拦截器统一处理HTTP错误。401 自动跳转登录页，500 弹出错误提示
3. **AI调用**：try-catch 包裹 + 最多3次重试 + 指数退避。单个文档提取失败不影响其他文档

### 综合提升类（Q76-Q100）

**Q76: 如果源文档和模板的字段名不一致怎么办？（比如模板写"联系电话"，文档中叫"手机号"）**

A: 三层解决：
1. **normalizeLabel()**：内置 15+ 字段映射，"联系电话"/"手机号"/"电话号码"都映射到 `phone`
2. **field_alias_dict 别名词典表**：数据库存储的可扩展映射，支持运行时添加新别名
3. **模糊包含匹配**："联系电话"包含"电话"，"手机号"包含个"手机"，通过子串匹配获得部分分数

**Q77: 系统能处理扫描版 PDF 吗？**

A: 目前不支持 PDF。PDF 结构复杂（二进制流、表格识别难度大），Apache PDFBox 提取表格准确率低。后续可集成 Tabula-Java（表格提取）或 OCR 引擎（扫描件识别）。当前优先保证 Word/Excel/TXT/MD 的高准确率。

**Q78: 你们和其他组做的项目有什么本质区别？**

A: 四个维度的差异化：
1. **模型自训练**：我们不只调 API，而是自己收集数据、训练微调了专用模型
2. **四端覆盖**：Web + Qt桌面 + 微信小程序 + MCP协议，不只是一个网页
3. **可解释性**：每个填入值都有完整的审计日志，不是黑盒
4. **工程完整度**：从一键部署脚本到 Docker 编排到企业级扩展方案，是可落地的系统

**Q79: 统计聚合怎么工作？给个例子**

A: 假设模板有个槽位叫"病例总数"，系统发现源 Excel 的表头也有"病例数"这一列（存了每个月的病例数）。系统检测到"总"这个关键词，自动对这列所有数值求 SUM，将求和结果填入模板。类似的："平均分" → AVG、"最高温" → MAX、"人员数量" → COUNT。

**Q80: 为什么要"一文档一调用"而不是批量发给 AI？**

A: 如果把5份文档的内容合成一个 Prompt，AI 容易出现"跨文档信息污染"——把 A 文档的甲方和 B 文档的金额搞混。一文档一调用虽然慢一点，但每个文档的提取结果是完全独立的，不会互相干扰。

**Q81: 如果用户上传了同一份文档两次会怎样？**

A: 重名文件会自动添加时间戳后缀（如 `合同.docx` → `合同_20240615143022.docx`），不会覆盖。每次上传都是独立的源文档记录，各自执行提取。如果提取结果重复，填表时的三重去重机制会处理。

**Q82: 微信小程序为什么要用 WebView 辅助模式？**

A: 某些复杂操作（如多文件选择、填表结果在线预览、ECharts 图表）在原生小程序中实现成本很高。通过 WebView 嵌入已有的 Web 版自动填表界面，复用 Web 端代码，开发效率更高。用户体验基本无感知。

**Q83: 项目的日志方案是什么？**

A: SLF4J + Logback，按服务分离日志文件（user-service.log / ai-service.log 等）。关键操作记 INFO 级别：
- AI 请求/响应（脱敏后）
- 填表决策过程（每个槽位的候选和最终决策）
- 文件上传/下载
- 错误和异常（ERROR级别）

**Q84: commonservice 公共模块包含什么？**

A: 被所有微服务共享的代码：
1. **实体类**：用户、文件、源文档、提取字段等 MyBatis-Plus 实体
2. **统一响应**：`Result<T>` 通用返回格式
3. **工具类**：文件操作、日期格式化、JSON 工具
4. **OSS 接口**：`OssService` 抽象层，支持本地存储和阿里云 OSS 切换

**Q85: 你说的"预提取架构"到底有多大的性能提升？**

A: 如果不预提取：上传 5 个模板，每个模板都要重新读取 20 份源文档 → 20 次 LLM 调用 × 5 = 100 次 LLM 调用，每次 5-30 秒。

有预提取：上传 20 份源文档时就完成了 20 次 LLM 调用。上传 5 个模板时只做规则匹配 → 0 次 LLM 调用（难例除外），每个模板 15-30 秒。

**总结**：从"100次LLM调用"降到"20次LLM调用"，且所有LLM调用都在后台异步完成，用户无需等待。

**Q86: 贪心兜底匹配和 LLM 终极兜底的置信度为什么不同？**

A: 贪心兜底（0.50）是纯规则匹配的结果，适用于"名称相似但初始匹配漏掉"的情况，准确度中等。LLM 终极兜底（0.65）是 AI 直接从原文中提取的结果，AI 理解了语义，准确度更高一些。但都低于正常匹配（可以到 0.90+），因为它们是"兜底"手段。

**Q87: 你们的微服务可以分开部署在不同机器上吗？**

A: 可以。每个服务通过 Nacos 注册自己的地址，Gateway 根据 Nacos 的服务列表路由。只要数据库、Redis、Nacos 可达，服务放在哪台机器上都行。ai-service 可以多实例部署，Gateway 自动负载均衡。

**Q88: 为什么前端在文档列表用 30 秒后改 8 秒轮询？**

A: LLM 提取通常 5-30 秒完成。前 30 秒高频轮询（3秒）是为了在提取完成后尽快更新 UI。30 秒后如果还没好，大概率是大文件或网络慢，放慢到 8 秒减少服务器压力。这是一个"快速响应vs资源消耗"的平衡。

**Q89: 你们怎么测试填表准确率的？**

A: 按比赛评价方法：
1. 上传测试文档样本集（5个docx + 3个md + 5个xlsx + 3个txt）
2. 等待所有提取完成
3. 逐个上传 5 个模板表格
4. 每个模板执行自动填表
5. 将填表结果与标准答案逐字段对比
6. 统计：正确填充 ÷ 应填充 = 准确率

**Q90: 模型量化（GGUF）是什么意思？对准确率有影响吗？**

A: 原始模型参数用 float16（16位浮点数）存储，量化就是把它压缩成 4 位整数（Q4_K_M），模型文件从 ~14GB 降到 ~4GB，推理速度更快。准确率损失非常小（约 1-2%），因为 4 位量化保留了大部分信息。这就像图片压缩——JPEG 比原图小很多，但肉眼几乎看不出区别。

**Q91: MLX 框架和 PyTorch 有什么区别？为什么用 MLX？**

A: MLX 是 Apple 专门为 Apple Silicon 设计的深度学习框架，对 Metal GPU 做了深度优化。在 M5 Max 上，MLX 的训练和推理效率比 PyTorch 高 30-50%（因为 PyTorch 对 Apple GPU 支持不完善）。如果在 NVIDIA GPU 服务器上，用 PyTorch 更合适。

**Q92: 你们的 QLoRA 训练了多长时间？**

A: 在 M5 Max 上，2106 条训练样本，约 3-4 个 epoch，总训练时间约 2-3 小时。QLoRA 因为只训练低秩矩阵参数很少，训练速度很快。如果全量微调，同样数据量可能需要几十小时。

**Q93: 如果评委问"你们怎么保证AI不会幻觉？"**

A: 我们通过系统设计来约束 AI 的输出：
1. **结构化输出**：要求 AI 以 JSON 格式输出，不接受自由文本
2. **候选选择题**：难例判定时 AI 只能从 5 个候选中选择，不能自由生成
3. **置信度机制**：低置信度自动拒填
4. **来源追溯**：每个值都绑定了源文档和原文，可以人工验证
5. **一文档一调用**：避免跨文档信息污染

简而言之：我们不让 AI 自由发挥，而是给它严格的"选择题"和"填空题"。

**Q94: 项目的商业前景如何？**

A: 已规划三级商业模式：
1. **社区版**（免费）：个人用户/学术团队，全功能开放，本地部署
2. **专业版**（¥798/年）：中小团队（5-50人），多用户协作、云端存储、优先技术支持
3. **企业版**（¥29800/年起）：大型组织，私有化部署、SSO集成、自定义词典、专属运维

核心卖点：支持完全私有化部署，数据不出企业内网，满足政务/金融/医疗等敏感行业的合规要求。

**Q95: 如果评委问"项目的不足之处"？**

A: 坦诚回答：
1. **不支持 PDF**：这是最常见的文档格式之一，目前未能支持
2. **大文件内存问题**：Apache POI DOM 模式在超大 Excel 时可能 OOM，需升级为流式处理
3. **测试覆盖率**：只有 77 个单元测试，缺乏 E2E 自动化测试
4. **模型轻量化**：7B 模型虽然不算大，但对于某些边缘设备仍然偏重

**Q96: 四个客户端的代码复用率高吗？**

A: 后端 API 是完全共享的——四个客户端调用相同的 RESTful 接口。但前端代码各自独立实现，因为：
1. Vue 3 代码不能直接用在小程序（WXML语法不同）
2. Qt C++ 和 JavaScript 是完全不同的语言
3. SSE 在三个平台的实现方式完全不同
4. 复用的部分是：API 接口文档、数据格式约定、业务逻辑流程

**Q97: MCP 服务目前支持哪些 AI 客户端？**

A: 任何支持 MCP 协议的 AI 客户端都可以，比如：Claude Desktop、Cursor IDE 的 AI Agent、自定义的 AI Agent 应用。MCP 是 Anthropic 提出的开放协议，正在被越来越多的 AI 工具支持。

**Q98: 你们团队是怎么分工的？**

A: （根据实际情况调整）
- 后端核心：AI 服务（信息提取 + 模板填充引擎） — 项目最复杂的部分
- 后端基础：用户服务 + 文件服务 + 网关配置 + 部署运维
- 前端：Vue 3 Web 端 + 微信小程序
- 桌面端：Qt C++ 桌面客户端
- 模型：数据收集 + 模型训练 + 部署
- 文档/测试：文档撰写 + 测试验证

**Q99: 如果有更多时间，你最想改进什么？**

A: 三个优先级最高的改进：
1. **PDF 支持**——这会让产品的适用场景扩大一倍
2. **流式 Excel 处理（SXSSFWorkbook）**——彻底解决大文件 OOM 问题
3. **知识库积累**——让系统从每次填表中学习，自动优化别名词典和评分权重

**Q100: 用一句话总结你们的项目**

A: DocAI 是一个基于自训练多模态大模型的智能文档处理平台，通过"预提取+规则优先+AI辅助"的分层架构，在 30 秒内将非结构化文档自动转化为结构化表格，覆盖 Web/桌面/小程序/AI Agent 四端，每个决策都可追溯审计。
