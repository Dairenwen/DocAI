## 二、前端模块详细流程

### 2.1 前端页面结构
```
App.vue (根组件)
├── Login.vue          ── 登录/注册页面
└── MainLayout.vue     ── 主布局框架（需认证）
    ├── Dashboard.vue     ── 工作台首页
    ├── DocumentList.vue  ── 文档管理
    ├── AutoFill.vue      ── 智能填表
    ├── AIChat.vue        ── AI对话
    └── AIGenerate.vue    ── AI写作
```

### 2.2 登录认证流程
```
用户输入用户名+密码
        │
        ▼
POST /api/v1/users/auth
        │
        ▼
┌──────────────────┐
│ user-service     │
│ UserController   │
│ .auth()          │
│   │              │
│   ▼              │
│ UserServiceImpl  │
│ .auth()          │
│   │              │
│   ├─ BCrypt验证密码（失败返回400，提示"用户名与密码不匹配"）
│   ├─ JwtUtil生成Token (HS256, 20h有效期)
│   └─ Redis存储Token（key: token:<jwt>）
└──────────────────┘
        │
        ▼
返回 { token, userId, username }
        │
        ▼
前端 localStorage 存储 token
```

### 2.3 文档上传与管理流程
```
用户在DocumentList.vue选择文件（支持docx/xlsx/txt/md四种格式）
        │
        ▼
POST /api/v1/source/upload (multipart/form-data)
        │
        ├─ 传输阶段 (进度0%-50%)
        │   uploadQueue进度条实时更新
        │
        ▼
┌─────────────────────────────────────────────────────┐
│ ai-service > ExtractionController.uploadAndExtract() │
│                                                      │
│  1. 文件保存到本地临时目录                              │
│  2. 上传至OSS (可选)                                  │
│  3. 创建source_documents记录 (status=parsing)         │
│  4. 调用 extractTextContent() 读取文件文本             │
│  5. 调用 callLlmExtract() 发送给LLM抽取               │
│  6. 解析LLM返回的JSON字段列表                          │
│  7. 标准化字段 + 存入extracted_fields表                │
│  8. 更新别名词典field_alias_dict                       │
│  9. 更新文档状态为 parsed                              │
└─────────────────────────────────────────────────────┘
        │
        ├─ 抽取阶段 (进度50%-100%)
        │  (完成/失败后2秒自动隐藏进度浮窗)
        │
        ▼
Pinia documentStore 刷新文档列表

智能轮询机制：
├── 上传后自动启用轻量级状态轮询（GET /source/documents/status）
├── 自适应间隔：前10次每3秒，之后每8秒
├── 最大轮询120次（约10分钟自动停止）
├── 检测到状态变化时才刷新完整文档列表
└── 全部文档解析完成后自动停止轮询

文档管理操作：
├── 文件预览：下载原文件在前端预览（txt/md直接显示，docx用mammoth渲染，xlsx用SheetJS渲染）
├── AI对话：关联源文档进入AI智能对话（需提取成功）
├── 下载：下载用户原始上传文件
├── 删除：删除源文档及关联的提取字段
├── 批量下载：勾选多个文档后一键批量下载
└── 批量删除：勾选多个文档后一键批量删除（调用 DELETE /source/batch）
```

## 三、后端微服务模块详解

### 3.1 用户服务 (user-service, Port 9001)

```
                    UserController
                    ├── POST /auth              → 登录认证
                    ├── POST /verification-code  → 发送验证码
                    ├── GET  /info              → 获取用户信息
                    ├── POST /change-password    → 修改密码
                    ├── POST /password/reset-by-email → 重置密码
                    └── POST /logout            → 退出登录

依赖链：
UserController → UserServiceImpl → UserMapper (MyBatis-Plus)
                                 → JwtUtil (Token管理)
                                 → RedisService (Token/验证码存储)
                                 → VerificationCodeService (6位验证码)
                                 → EmailService (发送验证码邮件)
```

### 3.2 文件服务 (file-service, Port 9003)

```
                    FilesController
                    ├── POST /files/upload/single → 上传Excel文件
                    ├── GET  /files/list         → 文件列表(分页)
                    ├── GET  /files/download      → 下载文件
                    ├── GET  /files/excel/preview/{id} → Excel预览(分页)
                    ├── GET  /files/excel/info/{id}    → Excel信息
                    ├── POST /files/restore/{id}       → 从OSS恢复数据
                    └── POST /files/delete              → 删除文件

核心流程（Excel上传）：
1. FileServiceImpl.upload()
   ├── 文件校验 (格式、大小50MB限制)
   ├── 上传至OSS
   ├── 创建files记录
   └── 调用ExcelToTableService
       ├── 读取Excel所有Sheet
       ├── 为每个Sheet动态建MySQL表
       ├── 批量插入数据 (1000行/批)
       ├── 记录file_table_mappings
       └── 记录field_mappings
```

│  │         逐Sheet逐行 → 纯文本                      │
│  ├─ txt  → Files.readString()                      │
│  └─ md   → Files.readString()                      │
└────────────────────────────────────────────────────┘
        │ 纯文本内容
        ▼
┌─ callLlmExtract() ─────────────────────────────────┐
│  构建抽取Prompt (含文件名、文档内容、输出Schema)       │
│        │                                            │
│        ▼                                            │
│  LlmService.generateText(prompt)                    │
│  → DashScopeProvider → 阿里云 qwen-turbo             │
│        │                                            │
│        ▼                                            │
│  解析返回的JSON                                      │
│  { fields: [ {field_key, field_name,                │

### 阶段3：模板槽位识别
```
模板文件 (xlsx/docx)
        │
        ▼
┌─ parseSlots() ──────────────────────────────────────┐
│                                                     │
│  xlsx → parseExcelSlots()                           │
│  ├─ 遍历所有Sheet                                    │
│  ├─ 遍历每个单元格                                    │
│  ├─ isLabelCell(): 判断是否为标签                     │
│  │   (长度2-20字、非纯数字、非说明文字)                 │
│  └─ 三类槽位识别:                                     │
│     ┌─────────────────────────────────────────┐      │
│     │规则1: 左边非空、右边空 → 右边是槽位         │      │
│     │  [项目名称] [______]                     │      │
│     │                                         │      │
│     │规则2: 上边非空、下边空 → 下边是槽位         │      │
│     │  [项目名称]                              │      │
│     │  [______]                               │      │
│     │                                         │      │
│     │规则3: 同单元格 "标签：____" → 内联槽位     │      │
│     │  [项目名称：________]                    │      │
│     └─────────────────────────────────────────┘      │
│                                                      │
│  docx → parseWordSlots()                             │
│  ├─ 遍历所有表格                                      │
│  ├─ 遍历每行每列                                      │
│  └─ 同样的三类槽位识别规则                             │
│                                                      │
│  统一输出: TemplateSlotEntity                         │
│  { label, context, position(JSON), expectedType }    │
└──────────────────────────────────────────────────────┘
        │
        ▼
数据库: template_slots 表
```

### 阶段4：候选字段召回
```
对每个TemplateSlot，从extracted_fields中召回Top-5候选：

TemplateSlot(label="项目名称")
        │
        ▼
┌─ recallCandidates() ────────────────────────────────┐
│  对每个ExtractedField计算5维得分:                      │
│                                                      │
│  1. alias_match_score (权重0.40)                     │
│     ├─ 精确匹配 label==fieldName → 1.0              │
│     ├─ 别名词典匹配 → 0.9                            │
│     ├─ 模糊包含匹配 → 0.5~0.7                       │
│     └─ 无匹配 → 0.0                                 │
│                                                      │
│  2. type_match_score (权重0.20)                      │
│     ├─ 类型完全匹配 → 1.0                            │
│     ├─ text类型兼容 → 0.3                            │
│     └─ 类型冲突 → 0.0                               │
│                                                      │
│  3. source_vote_score (权重0.20)                     │
│     └─ 多文档出现相同值 → votes/maxVote              │
│                                                      │
│  4. context_match_score (权重0.10)                   │
│     └─ 上下文关键词匹配 → 0.8                        │
│                                                      │
│  5. field_confidence (权重0.10)                      │
│     └─ LLM抽取时的置信度                             │
│                                                      │
│  candidate_score = 0.40*alias + 0.20*type            │
│                  + 0.20*vote + 0.10*context           │
│                  + 0.10*confidence                    │
│                                                      │
│  排序取Top-5, 过滤score < 0.1                        │
└──────────────────────────────────────────────────────┘
        │ List<CandidateResult> (Top-5)
        ▼
```

### 阶段5：难例判定（LLM）
```
                 Top-5 候选
                     │
                     ▼
┌─ makeDecision() ────────────────────────────────────┐
│                                                      │
│  是否需要调用LLM？(满足任一条件即调用)                  │
│  ┌─────────────────────────────────────────────┐     │
│  │ 1. Top1 - Top2 分差 < 0.15                  │     │
│  │ 2. Top1 分数 < 0.75                         │     │
│  │ 3. 候选值类型冲突                            │     │
│  │ 4. 多文档给出不同值且分差<0.20               │     │
│  │ 5. 别名词典命中分 < 0.5                      │     │
│  └─────────────────────────────────────────────┘     │
│                                                      │
│  若需要 → callLlmJudge()                             │
│  ├─ 构建Prompt (仅传slot信息 + Top-5候选及sourceText) │
│  ├─ LlmService.generateText() → qwen-plus            │
│  ├─ 解析JSON响应                                     │
│  │   { selectedFieldId, selectedValue,               │
│  │     modelConfidence, reason }                     │
│  └─ 若modelConfidence < 0.65 → 拒填                  │
│                                                      │
│  若不需要 → 直接取Top-1候选                           │
└──────────────────────────────────────────────────────┘
```

### 阶段6：置信度融合与拒填
```
               FillDecisionEntity
                     │
                     ▼
┌─ applyConfidenceThreshold() ─────────────────────────┐
│                                                       │
│  final_confidence 计算:                               │
│  ├─ 若调用了LLM:                                     │
│  │   = 0.70 * candidate_score + 0.30 * model_conf    │
│  └─ 若纯规则:                                        │
│      = candidate_score                                │
│                                                       │
│  决策阈值:                                            │
│  ┌───────────────────────────────────────────────┐   │
│  │ ≥ 0.70  → ✅ 直接填写（高置信）                │   │
│  │ 0.20~0.70 → ⚠️ 填写，标记建议人工复核         │   │
│  │ < 0.20  → ❌ 拒绝填写，保留空白               │   │
│  └───────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────┘
```

## 五、AI对话模块流程

### 5.1 对话会话管理

对话记录通过服务端持久化存储（MySQL），支持跨浏览器/设备访问：
- 会话列表管理：`GET/POST/PUT/DELETE /api/v1/ai/conversations`
- 消息管理：`GET/POST /api/v1/ai/conversations/{id}/messages`
- 会话时间仅在发送/接收消息时更新（addMessage 时触发 updatedAt 更新）

### 5.2 AI流式对话流程

### 5.3 停止生成功能

前端支持在两个阶段中止AI生成：
1. **后端请求阶段**（loading状态）：通过 AbortController 中止 SSE fetch 请求
2. **打字动画阶段**（isStreaming状态）：停止定时器，立即展示已接收的完整内容

停止按钮替代发送按钮显示，点击后立即恢复输入状态。

### 5.4 AI文档在线编辑与导出

用户在对话中提出文档修改需求（如"把标题改为XXX"、"删除第三段"、"润色全文"），系统自动识别编辑意图并执行：

```
用户输入 "将标题修改为《2024年度报告》"
        │
        ▼
handleSourceDocChatFlow()
        │
        ├── 1. 编辑意图检测（关键词匹配）:
        │   修改/编辑/删除/增加/添加/替换/改为/改成/更新/移除/插入/润色
        │
        ├── 2. 读取源文档全文内容
        │
        ├── 3. 构建编辑Prompt → LLM返回修改后完整文档
        │
        ├── 4. 自动保存修改后文档:
        │   ├── .docx → XWPFDocument 生成Word
        │   ├── .md   → Files.writeString
        │   └── .txt  → Files.writeString
        │   保存路径: {tmpdir}/docai_edited/{原文件名_edited.ext}
        │
        ├── 5. 返回SSE响应:
        │   {isModificationRequest: true, modifiedExcelUrl: 下载URL}
        │
        └── 6. 前端弹出下载链接，用户点击即可导出
```

**API接口**：
- `POST /api/v1/ai/documents/{docId}/apply-edit` — 手动触发编辑保存
- `GET /api/v1/ai/documents/edited/{fileName}` — 下载修改后的文档

**安全措施**：文件名校验禁止路径遍历（`..`、`/`、`\`等非法字符）。
