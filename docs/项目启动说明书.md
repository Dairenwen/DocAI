## 一、环境要求

### 1.1 软件依赖

| 软件 | 最低版本 | 用途 | 安装验证命令 |
|------|---------|------|-------------|
| JDK | 17+ | 后端运行环境 | `java -version` |
| Maven | 3.8+ | Java 项目构建 | `mvn -version` |
| Node.js | 18+ | 前端构建 | `node -v` |

| 环境 | CPU | 内存 | 磁盘 | 说明 |
|------|-----|------|------|------|
| 开发/单机 | 4核+ | 8GB+ | 20GB+ | 所有服务运行在同一台机器 |
| 生产-最低 | 4核+ | 16GB+ | 50GB+ | 中间件 + 微服务同机 |
| 生产-推荐 | 8核+ | 32GB+ | 100GB+ | 微服务与中间件分离部署 |

| 端口 | 服务 | 说明 |
|------|------|------|
| 8080 | Nginx | 前端入口 + API 反向代理 |

| 9001 | user-service | 用户认证服务 |
| 9002 | ai-service | AI 核心服务（抽取/填表/对话） |
| 9003 | file-service | 文件管理服务 |

| 8848/9848/9849 | Nacos | 配置/注册中心 |

## 三、快速启动（单机部署）

### 3.1 Windows 一键启动

```powershell
cd docai-pro
.\start-lite-windows.ps1
```

脚本自动执行以下步骤：
1. **环境检测**：检查 Docker / Maven / Node / Java 可用性
2. **后端构建**：`mvn -DskipTests clean package`（自动停止旧 Java 进程）
3. **前端构建**：`npm install` + `npm run build`
4. **前端部署**：同步 dist 到 Nginx 静态目录
5. **中间件启动**：Docker Compose 启动 MySQL 8.4 / Redis 7.0 / Nacos 2.2 / Nginx 1.24
6. **微服务启动**：依次启动 user-service(9001) → file-service(9003) → ai-service(9002) → gateway(18080)

### 3.2 Linux 一键启动

```bash
cd docai-pro
chmod +x start-lite.sh
export DOC_DASHSCOPE_API_KEY="your-api-key"
./start-lite.sh
```

### 3.3 快速重启（跳过编译）

已完成首次构建后，可使用快速重启脚本，跳过 Maven/npm 构建步骤：

```powershell
cd docai-pro
.\start-services-only.ps1
```

> 前提：JAR 包已存在、Docker 中间件已运行。

### 3.4 停止所有服务

```powershell
# Windows
cd docai-pro
.\stop-lite-windows.ps1

# Linux
cd docai-pro
./stop-lite.sh
```

### 3.5 启动后验证

| 验证项 | 访问地址 | 期望结果 |
|--------|---------|---------|
| 前端页面 | http://localhost:8080 | 显示登录页面 |
| API 网关 | http://localhost:18080 | 网关服务响应 |
| Nacos 控制台 | http://localhost:8848/nacos | 用户名/密码：nacos/nacos |
| 登录测试 | POST http://localhost:8080/api/v1/users/auth | 返回 JWT Token |

首次访问自动注册：输入任意用户名和密码即可自动注册并登录。

## 六、企业生产部署方案

### 6.1 单服务器部署（最简方案）

适用于小团队或资源有限的场景，所有服务运行在一台服务器上：

```
┌──────────────────────────────────────────────────┐
│              单服务器 (推荐 8核 16G+)                │
│                                                    │
│  ┌─────────┐ ┌────────────────────────────────┐  │
│  │ Docker   │ │ Java 微服务                     │  │
│  │ MySQL    │ │ user-service (9001)             │  │
│  │ Redis    │ │ file-service (9003)             │  │
│  │ Nacos    │ │ ai-service   (9002)             │  │
│  │ Nginx    │ │ gateway      (18080)            │  │
│  └─────────┘ └────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

直接使用 `start-lite.sh` 一键部署，唯一区别是设置 `PUBLIC_HOST` 环境变量为公网 IP。

### 6.2 双服务器部署（推荐方案）

将中间件和应用服务分离，提高稳定性和性能：

```
┌─────────────────────────────┐     ┌─────────────────────────────────┐
│    服务器A: 中间件层          │     │    服务器B: 应用层                 │
│    (4核 8G)                  │     │    (8核 16G)                     │
│                              │     │                                  │
│  ┌──────────────────────┐   │     │  ┌───────────────────────────┐  │
│  │ Docker Compose        │   │     │  │ Nginx (8080)              │  │
│  │  - MySQL 8.4 (3306)  │   │◄────┤  │ Gateway (18080)           │  │
│  │  - Redis 7.0 (6379)  │   │     │  │ user-service (9001)       │  │
│  │  - Nacos 2.2 (8848)  │   │     │  │ file-service (9003)       │  │
│  └──────────────────────┘   │     │  │ ai-service (9002)         │  │
└─────────────────────────────┘     │  └───────────────────────────┘  │
                                     └─────────────────────────────────┘
```

**部署步骤**：

**服务器 A（中间件）**：
```bash
# 1. 安装 Docker
# 2. 上传 deploy/ 目录
cd deploy
# 修改 docker-compose-mid.yml 中的端口绑定为 0.0.0.0
docker compose -f docker-compose-mid.yml up -d
```

**服务器 B（应用）**：
```bash
# 1. 安装 JDK 17、Maven、Node.js
# 2. 上传整个项目
# 3. 修改启动脚本中的数据库/Redis/Nacos 连接地址指向服务器 A 的 IP
# 4. 编译并启动
cd docai-pro
mvn -DskipTests clean package
cd ../docai-frontend && npm install && npm run build
# 5. 手动启动各微服务，将 127.0.0.1 改为服务器 A 的内网 IP
java -jar user-service/target/user-service-1.0.0.jar \
  --spring.datasource.url="jdbc:mysql://<服务器A-IP>:3306/docai?..." \
  --spring.redis.host=<服务器A-IP> \
  ...
```

### 6.3 多服务器分布式部署（企业级方案）

适用于高并发、高可用的企业级场景：

```
┌─────────────────────────────────────────────────────────────────┐
│                        负载均衡层                                 │
│              SLB / Nginx (公网 IP + HTTPS)                       │
└──────────────┬──────────────┬──────────────┬────────────────────┘
               │              │              │
┌──────────────▼──┐ ┌────────▼────────┐ ┌───▼──────────────┐
│   网关节点 ×2    │ │   网关节点 ×2    │ │   网关节点 ×2    │
│   Gateway       │ │   Gateway       │ │   (预留扩展)     │
│   Nginx         │ │   Nginx         │ │                  │
└────────┬────────┘ └────────┬────────┘ └──────────────────┘
         │                   │
┌────────▼───────────────────▼────────────────────────────────────┐
│                       微服务层                                    │
│                                                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ user-service │  │ file-service │  │ ai-service ×2~3      │  │
│  │ ×1~2 实例    │  │ ×1~2 实例    │  │ (最消耗资源、可多实例) │  │
│  └──────────────┘  └──────────────┘  └──────────────────────┘  │
└─────────────────────────────┬────────────────────────────────────┘
                              │
┌─────────────────────────────▼────────────────────────────────────┐
│                       数据层                                      │
│                                                                   │
│  ┌────────────────┐  ┌────────────┐  ┌────────────────────────┐ │
│  │ MySQL 主从     │  │ Redis 主从  │  │ Nacos 集群 (3节点)      │ │
│  │ (主写从读)     │  │ + 哨兵     │  │ (服务注册 + 配置管理)   │ │
│  └────────────────┘  └────────────┘  └────────────────────────┘ │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │              共享存储 (阿里云 OSS / NAS)                      │ │
│  │         源文档 + 模板文件 + 填充结果跨节点共享                  │ │
│  └─────────────────────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────────────────────┘
```

**关键配置要点**：

1. **网关层**：
   - 部署 2+ 个 Gateway 实例，前端通过 SLB（如阿里云 SLB、Nginx Plus）负载均衡
   - 配置 HTTPS 证书，所有外部访问走 443 端口
   - Gateway 的路由 URI 从固定 IP 改为 Nacos 服务发现（`lb://ai-service`）

2. **微服务层**：
   - `ai-service` 是最消耗资源的服务（LLM 调用），建议 2~3 个实例
   - `user-service` / `file-service` 通常 1~2 个实例即可
   - 每个实例通过 Nacos 注册，Gateway 自动发现并负载均衡
   - 启用 Nacos 配置管理，实现配置热更新无需重启

3. **数据层**：
   - MySQL 主从复制（1主1从或1主2从），读写分离
   - Redis 主从 + Sentinel 哨兵模式，自动故障转移
   - Nacos 3 节点集群，保证 Raft 协议高可用

4. **存储层**：
   - 本地文件存储（`local-oss/`）迁移到阿里云 OSS 或 NAS 共享挂载
   - 配置 `DOC_OSS_ENDPOINT` 等 OSS 环境变量启用云存储
   - 确保所有微服务实例访问同一存储路径

5. **启用 Nacos 服务发现**（修改启动参数）：
   ```bash
   # 将以下参数从 false 改为 true
   --spring.cloud.nacos.discovery.enabled=true
   --spring.cloud.nacos.config.enabled=true
   --spring.cloud.nacos.discovery.server-addr=<nacos-cluster-addr>
   
   # Gateway 路由改为服务发现模式
   --spring.cloud.gateway.routes[0].uri=lb://user-service
   ```

6. **可观测性建设**：
   - 接入 Prometheus + Grafana 监控 JVM 和业务指标
   - 接入 ELK（Elasticsearch + Logstash + Kibana）统一日志收集
   - Spring Boot Actuator 暴露 `/actuator/health` 健康检查端点

### 6.4 企业部署核验清单

| 检查项 | 说明 |
|--------|------|
| ✅ HTTPS 证书配置 | Nginx 配置 SSL 证书，禁止 HTTP 明文传输 |

## 七、常见问题排查

### 7.1 启动失败

| 问题 | 解决方案 |
|------|---------|
| `Unresolved compilation problems` | 执行 `mvn -DskipTests clean package` 重新编译 |
| Maven `clean` 失败（JAR 被占用） | 先运行 `stop-lite-windows.ps1` 停止旧进程 |
| Docker 容器启动失败 | 检查端口是否被占用：`netstat -ano \| findstr :3306` |
| 前端构建失败 | 删除 `node_modules` 后重新 `npm install` |
| MySQL 初始化未执行 | 清除 `docai-pro/data/docaidata/mysql/data/` 后重启容器 |
| Nacos 启动超时 | 等待 30 秒后再启动 Gateway，Nacos 需要初始化时间 |

### 7.2 运行时问题

| 问题 | 解决方案 |
|------|---------|
| AI 提取超时 | 检查 DashScope API Key 是否有效，网络是否能访问阿里云 |
| 填表结果全空 | 确认源文档已提取成功（uploadStatus=parsed） |
| 文件上传 413 | Nginx 配置 `client_max_body_size 200m` |
| SSE 连接断开 | 检查 Nginx `proxy_read_timeout` 是否 ≥300s |
| 跨域错误 | 检查 Nginx 和 Gateway 的 CORS 配置 |

### 7.3 性能调优

| 服务 | 默认内存 | 建议调整 | 说明 |
|------|---------|---------|------|
| user-service | -Xmx256m | 256m~512m | 轻量级服务 |
| file-service | -Xmx256m | 256m~512m | 大文件上传时可调高 |
| ai-service | -Xmx768m | 768m~2048m | LLM 调用 + POI 处理 |
| gateway-service | -Xmx256m | 256m~512m | 路由转发 |

## 八、Ngrok 内网穿透配置

### 8.1 概述

当服务器配置不足或无法部署到公网服务器时，可使用 [Ngrok](https://ngrok.com/) 内网穿透工具将本地服务暴露到公网，让外部设备（如手机、其他电脑）通过固定域名访问。

### 8.2 安装 Ngrok

1. 前往 [Ngrok 官网](https://ngrok.com/) 注册账号
2. 下载 Windows 版本并解压到任意目录（如 `C:\ngrok\`）
3. 添加 Ngrok 到系统 PATH（可选）
4. 认证 Ngrok 客户端：
   ```powershell
   ngrok config add-authtoken <your-auth-token>
   ```
   > Auth Token 在 Ngrok Dashboard → Your Authtoken 页面获取

### 8.3 配置固定域名

Ngrok 免费版提供随机域名（每次启动变化），付费版（如 Basic 计划 $8/月）支持固定自定义域名。

**方案一：免费版（随机域名）**

```powershell
# 穿透 Nginx 端口（推荐，一个隧道覆盖前端+API）
ngrok http 8080
```

启动后会显示类似 `https://xxxx-xxx-xxx.ngrok-free.app` 的公网 URL。

**方案二：Ngrok 免费静态域名（推荐）**

Ngrok 现已为免费用户提供 1 个免费的静态域名：

1. 登录 [Ngrok Dashboard](https://dashboard.ngrok.com/)
2. 进入 **Domains** 页面 → **Create Domain**
3. 获取分配的免费静态域名（如 `your-name.ngrok-free.app`）
4. 使用静态域名启动隧道：
   ```powershell
   ngrok http --domain=your-name.ngrok-free.app 8080
   ```

**方案三：付费版（完全自定义域名）**

```powershell
# 使用自定义域名
ngrok http --domain=docai.your-company.com 8080
```

> 需要在域名 DNS 中添加 CNAME 记录指向 Ngrok 提供的地址。

### 8.4 Ngrok 配置文件（多隧道）

创建 Ngrok 配置文件 `ngrok.yml`：

```yaml
version: "2"
authtoken: <your-auth-token>
tunnels:
  docai-web:
    addr: 8080
    proto: http
    domain: your-name.ngrok-free.app    # 使用固定域名
```

启动：
```powershell
ngrok start --config ngrok.yml --all
```

### 8.5 使用流程

1. **先启动 DocAI 服务**：确保 `http://localhost:8080` 可正常访问
2. **启动 Ngrok 隧道**：
   ```powershell
   ngrok http --domain=your-name.ngrok-free.app 8080
   ```
3. **获取公网地址**：Ngrok 命令行会显示公网 URL
4. **外部设备访问**：
   - 浏览器直接访问 `https://your-name.ngrok-free.app`
   - 小程序配置 `baseURL` 为 `https://your-name.ngrok-free.app/api/v1`
5. **验证连通性**：在外部设备上测试登录和文档上传功能

### 8.6 注意事项

| 注意项 | 说明 |
|--------|------|
| HTTPS 自动配置 | Ngrok 自动为隧道提供 HTTPS 证书，无需手动配置 |
| 免费版提示页 | 免费计划首次访问会显示 Ngrok 品牌提示页，点击 "Visit Site" 继续 |
| 带宽限制 | 免费计划每月 1GB 流量，付费无限制 |
| 连接数限制 | 免费计划最多 40 个连接/分钟 |
| 稳定性 | 建议生产使用付费版；开发/演示用免费版足够 |
| 防火墙 | 确保本机防火墙允许 Ngrok 出站连接 |
| SESSION_AFFINITY | 如部署多实例，需配置会话亲和性 |

### 8.7 持久化运行（后台启动）

使用 PowerShell 后台运行 Ngrok：

```powershell
# 后台启动
Start-Process ngrok -ArgumentList "http","--domain=your-name.ngrok-free.app","8080" -WindowStyle Hidden

# 查看隧道状态
Invoke-RestMethod http://127.0.0.1:4040/api/tunnels | ConvertTo-Json

# 停止
Get-Process ngrok | Stop-Process
```

或使用 Ngrok 的 Web 管理界面：`http://127.0.0.1:4040`

## 九、SMTP 邮件服务配置

### 9.1 QQ 邮箱配置

1. 登录 QQ 邮箱 → 设置 → 账户 → 开启 SMTP 服务
2. 获取授权码（16位字母）
3. 配置环境变量：
   ```powershell
   $env:DOC_SMTP_HOST = "smtp.qq.com"
   $env:DOC_SMTP_PORT = "587"
   $env:DOC_SMTP_USER = "your-qq@qq.com"
   $env:DOC_SMTP_AUTH_CODE = "your-16-char-auth-code"
   ```

### 9.2 163 邮箱配置

```powershell
$env:DOC_SMTP_HOST = "smtp.163.com"
$env:DOC_SMTP_PORT = "465"
$env:DOC_SMTP_USER = "your-email@163.com"
$env:DOC_SMTP_AUTH_CODE = "your-auth-code"
```
